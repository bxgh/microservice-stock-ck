## 0. 为什么需要这份补充规范

原规范 v2 覆盖"给定分组后怎么算",但**完全没有覆盖"分组本身怎么定义和演化"**。
这在工程上是一个真实缺口:

- `layer_id` 是 1-6 的整数,但每一层里"哪些股票在一起"没有规范
- `DislocationEvent` 只处理**已有层级间**的异常,不处理**新主题的浮现**
- SW 行业和 THS 概念直接使用会引入大量噪音(成员 <10、高度重叠、蹭热点)
- 系统无法识别"资金已经在形成新耦合但现有分类还没反应"的情形

本补充规范解决这三个问题:
1. 规范化六层架构的分组定义与维护
2. 引入**动态主题发现模块**作为分组的挑战者和新主题侦察雷达
3. 将主题发现的输出接入现有的 CCI 计算管线与告警系统

**与原规范的关系:**
- 不修改原规范任何定义(ρ̄、CCI、α/β/γ/δ、层数等)
- 只**扩展**数据模型、API、告警、开发路线
- 新增的所有功能都是**可关闭的插件**,不影响核心 CCI 管线可用性

## 1. 六层架构的分组定义规范

### 1.1 层级语义正式定义

| 层 | 语义 | 粒度 | 分组来源 | 更新频率 |
|---|---|---|---|---|
| L1 | 全市场 | ~5000 只(剔除后约 3500) | 全体 A 股 - 过滤规则 | 静态 |
| L2 | 风格/规模 | 3-6 个组 | 市值分层 + 风格因子 | 季度 |
| L3 | 行业 | 28-31 个组 | SW 一级 | 静态(SW 调整时跟随) |
| L4 | 主题 | 30-80 个组 | **自建主题库** | 月度 |
| L5 | 产业链环节 | L4 下二级 | **人工维护(基于研报 + 10 步评估)** | 月度 |
| L6 | 核心/跟风簇 | L4 内部 2-3 簇 | **动态社区发现输出** | 每日/每周 |

### 1.2 L1 全市场过滤规则(必须实施)

## 3. 动态主题发现模块

### 3.1 模块定位

这是一个**独立子系统**,与 CCI 核心管线解耦:
- 输入:L1 全市场收益率矩阵
- 输出:候选新主题、社区结构变化告警、L6 核心/跟风簇
- 输出消费者:(a) 主题库维护者(人工审核)、(b) CCI 管线的 L6 计算

### 3.2 算法流水线

```
流动性股票池 (~500)
    ↓
日度对数收益率矩阵 R (T × N)
    ↓
60 日滚动 Pearson 相关 → 相关矩阵 C (N × N)
    ↓
RMT 去噪:剔除市场因子 + 噪音特征值
    ↓
阈值化:ρ_ij < 0.3 的边删除
    ↓
构建加权无向网络 G
    ↓
Leiden 社区发现(分辨率扫描 γ ∈ [0.5, 1.5])
    ↓
输出:community_assignments[stock] → community_id
    ↓
社区对齐分析:与 L3/L4 主题库比对
    ↓
生成告警:新簇浮现 / 社区重组 / 边界漂移
```

### 3.4 关键算法代码

**RMT 去噪(核心):**

```python
# backend/src/cci_monitor/discovery/rmt.py

import numpy as np

def rmt_denoise_correlation(
    corr: np.ndarray,
    T: int,
    remove_market_factor: bool = True,
) -> np.ndarray:
    """
    Marčenko-Pastur 去噪.

    保留的部分:
    - (可选)市场因子 = 最大特征值对应的投影
    - 落在 [λ-, λ+] 外的显著特征值

    剔除:噪音特征值(位于 MP 分布内)

    参数:
        corr: (N, N) 相关矩阵
        T: 用于估计相关的样本数(天数)
        remove_market_factor: 是否同时剔除市场因子(用于"纯主题"分析)

    返回:
        denoised: (N, N) 去噪后的相关矩阵
    """
    N = corr.shape[0]
    q = T / N  # 必须 q > 1 才能估计,建议 T >= 2N

    # MP 分布的上下界
    lambda_plus = (1 + 1/np.sqrt(q))**2
    lambda_minus = (1 - 1/np.sqrt(q))**2

    # 特征分解
    eigvals, eigvecs = np.linalg.eigh(corr)

    # 保留特征值的 mask
    significant = eigvals > lambda_plus
    if remove_market_factor:
        # 剔除最大特征值(市场因子)
        max_idx = np.argmax(eigvals)
        significant[max_idx] = False

    # 用保留的特征值重建
    kept_eigvals = np.where(significant, eigvals, 0)
    denoised = eigvecs @ np.diag(kept_eigvals) @ eigvecs.T

    # 重新标准化对角线为 1
    d = np.sqrt(np.diag(denoised))
    d[d < 1e-10] = 1.0
    denoised = denoised / np.outer(d, d)

    return denoised
```

**Leiden 社区发现:**

```python
# backend/src/cci_monitor/discovery/community.py

import igraph as ig
import leidenalg
import numpy as np

def find_communities(
    corr: np.ndarray,
    stocks: list[str],
    threshold: float = 0.3,
    resolution: float = 1.0,
    seed: int = 42,
    n_iterations: int = 10,
) -> tuple[dict[str, int], float]:
    """
    基于相关矩阵的 Leiden 社区发现.

    参数:
        corr: (N, N) 去噪后的相关矩阵
        stocks: 长度 N 的股票代码列表
        threshold: 边权重阈值(低于则剔除)
        resolution: Leiden 分辨率参数
        seed: 随机种子
        n_iterations: 跑多次取最优

    返回:
        assignments: {stock_code: community_id}
        modularity: 最优分区的模块度

    性能目标: N=500 时 < 10 秒
    """
    N = corr.shape[0]

    # 构建 igraph 网络
    edges = []
    weights = []
    for i in range(N):
        for j in range(i+1, N):
            if corr[i, j] > threshold:
                edges.append((i, j))
                weights.append(float(corr[i, j]))

    if len(edges) == 0:
        # 所有边都被过滤,每个节点自成一簇
        return {s: i for i, s in enumerate(stocks)}, 0.0

    g = ig.Graph(n=N, edges=edges)
    g.es['weight'] = weights

    # 跑 n_iterations 次,取模块度最大的
    best_partition = None
    best_modularity = -np.inf

    for i in range(n_iterations):
        partition = leidenalg.find_partition(
            g,
            leidenalg.RBConfigurationVertexPartition,
            resolution_parameter=resolution,
            weights='weight',
            seed=seed + i,
        )
        if partition.modularity > best_modularity:
            best_modularity = partition.modularity
            best_partition = partition

    assignments = {
        stocks[node]: community_id
        for community_id, nodes in enumerate(best_partition)
        for node in nodes
    }

    return assignments, best_modularity


def resolution_sweep(
    corr: np.ndarray,
    stocks: list[str],
    resolutions: list[float] = None,
) -> list[dict]:
    """
    多分辨率扫描,对应不同粒度的分层.

    粗 γ=0.5 ≈ 风格/大板块  (接近 L2)
    中 γ=1.0 ≈ 主题          (接近 L4)
    细 γ=1.5 ≈ 产业链子簇    (接近 L5-L6)
    """
    if resolutions is None:
        resolutions = [0.5, 0.8, 1.0, 1.3, 1.5]

    results = []
    for gamma in resolutions:
        assignments, Q = find_communities(corr, stocks, resolution=gamma)
        n_communities = len(set(assignments.values()))
        results.append({
            "resolution": gamma,
            "n_communities": n_communities,
            "modularity": Q,
            "assignments": assignments,
        })
    return results
```

### 3.5 性能要求

| 步骤 | N=500 目标 | N=1000 目标 |
|---|---|---|
| 相关矩阵计算 | < 1 秒 | < 4 秒 |
| RMT 去噪 | < 0.5 秒 | < 3 秒 |
| Leiden(单次) | < 2 秒 | < 8 秒 |
| Leiden(10 次取最优) | < 10 秒 | < 60 秒 |
| 分辨率扫描(5 个 γ) | < 30 秒 | < 200 秒 |

## 4. 社区 vs 主题库对齐分析

动态社区的**最大价值不是"取代主题库",而是"挑战主题库"**。对齐分析输出四种状态,每种对应不同告警。

### 4.1 四种对齐状态

```python
class AlignmentStatus(str, Enum):
    CONSISTENT       = "consistent"        # 社区 ≈ 某个已有主题 → 验证主题库
    EMERGING         = "emerging"          # 跨行业的统计簇,无对应主题 → 新主题候选 ⚠️
    FRAGMENTING      = "fragmenting"       # 主题内部被劈成多簇 → 主题边界需调整 ⚠️
    DISSOLVING       = "dissolving"        # 主题成员散落到多个社区 → 主题可能退潮 ⚠️
```

### 4.2 对齐度量

```python
# backend/src/cci_monitor/discovery/alignment.py

from collections import Counter

def compute_alignment(
    community_assignments: dict[str, int],
    theme_memberships: dict[str, list[str]],  # {theme_id: [stocks]}
) -> list[AlignmentResult]:
    """
    对每个社区、每个主题,计算对齐度.

    关键指标:
    - Jaccard(community, theme): |交集| / |并集|
    - Precision(community → theme): |交集| / |community|
    - Recall(theme → community): |交集| / |theme|
    """

    results = []

    # 按社区聚合股票
    communities: dict[int, set[str]] = {}
    for stock, cid in community_assignments.items():
        communities.setdefault(cid, set()).add(stock)

    for cid, c_members in communities.items():
        if len(c_members) < 10:  # 小社区跳过
            continue

        # 找和这个社区最重叠的主题
        best_theme = None
        best_jaccard = 0.0
        for theme_id, t_members in theme_memberships.items():
            t_set = set(t_members)
            jaccard = len(c_members & t_set) / len(c_members | t_set)
            if jaccard > best_jaccard:
                best_jaccard = jaccard
                best_theme = theme_id

        # 判定状态
        if best_jaccard > 0.6:
            status = AlignmentStatus.CONSISTENT
        elif best_jaccard < 0.2:
            # 检查是否跨多个行业
            industry_spread = count_industries(c_members)
            if industry_spread >= 3:
                status = AlignmentStatus.EMERGING
            else:
                status = AlignmentStatus.CONSISTENT  # 只是被现有主题漏收
        else:
            status = AlignmentStatus.FRAGMENTING

        results.append(AlignmentResult(
            community_id=cid,
            community_size=len(c_members),
            best_theme_id=best_theme,
            jaccard=best_jaccard,
            status=status,
            member_stocks=sorted(c_members),
        ))

    return results
```

### 4.3 告警语义

每种状态对应的告警动作:

| 状态 | 动作 | 通知对象 |
|---|---|---|
| CONSISTENT | 无告警,记录日志 | - |
| EMERGING | **发送"新主题候选"通知** | 人工审核队列 |
| FRAGMENTING | **发送"主题分裂"预警** | 主题库维护者 |
| DISSOLVING | **发送"主题退潮"预警** | 交易决策者 |

## 9. 给 AI Coding Agent 的关键提示

### 9.1 不要做的事

- **不要让发现模块污染 CCI 核心管线**。L4/L5 的 ρ̄ 必须仍然基于**稳定主题库**,不是基于每日变动的社区。社区只影响 L6 和候选新主题告警。
- **不要自动把候选新主题加入主题库**。必须经过人工审核(`promote` API 调用)。
- **不要在 Leiden 中使用 resolution=1.0 作为唯一值**。必须支持多分辨率,至少提供 `{0.8, 1.0, 1.3}` 三档。
- **不要忽略模块度 Q**。Q < 0.3 意味着社区结构不显著,该日的发现结果应标记为"低置信度"。

### 9.2 必须做的事

- **group_hash 必须严格计算**。任何 CCI 记录都必须能追溯到当天的精确分组。
- **主题 retire 是软删除**。归档主题的历史 ρ̄ 必须保留。
- **告警去重**。同一个社区连续多天被识别为 EMERGING,只发一次告警(除非社区成员显著变化)。
- **默认关闭发现模块**。通过配置开关 `settings.discovery.enabled` 控制。核心 CCI 管线不依赖它。

### 9.3 A 股特殊处理

- **涨跌停日**的 NaN 处理:如果个股当日涨跌停导致收益率极值,在计算相关矩阵时该值权重应降低(或直接视为缺失)。
- **停牌处理**:停牌期间不贡献任何相关性,在 rolling window 中应跳过而非填零。
- **北交所**:默认排除(流动性差会污染相关矩阵)。需要时通过 `MarketUniverseFilter.beijing_exchange=True` 开启。
- **指数成分调整**:SW 行业或 HS300/ZZ500 成分调整时,L3 分组要跟随调整,同时在 `theme_members.effective_to` 中记录旧成员的退出时间。

## 11. 附录:理论依据速查

**为什么是 Leiden 而不是 Louvain**:Louvain 不保证社区内部连通(Traag et al. 2019),在金融相关矩阵上这个问题经常出现,会把散沙伪装成一簇。Leiden 通过精化阶段强制连通,是严格改进。

**为什么要 RMT 去噪**:Laloux et al. (1999) / Plerou et al. (1999) 证明金融相关矩阵的大部分特征值落在 Marčenko-Pastur 随机矩阵分布内,不含信号。不去噪会让市场因子和噪音污染社区发现。

**为什么分辨率要扫描**:单一分辨率对应单一粒度,但你的六层架构本质上要求多粒度。Reichardt-Bornholdt 2006 的分辨率参数族正是为此设计。

**为什么 L6 必须由算法产生**:L6 是"核心簇 vs 跟风簇"。这个区分是动态的、依赖数据的,人工维护会在一两周内过时。社区发现在每个 L4 主题内部跑一次就能产生这个区分。
