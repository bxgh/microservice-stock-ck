# E3: 巡检模块独立部署(P0)

> 这是原文重点,但要明确边界:巡检只做"发现问题",修复交给 E6。

## Story E3-S1: 部署独立巡检节点

* **物理隔离**:不要和日常采集服务跑在同一台机器
* 推荐放在内网 NAS / 自有 PC / 性能稍好的边缘节点
* 调度时间避开交易时段和日常采集窗口(建议每日凌晨 2:00–5:00)

## Story E3-S2: 实现三类核心巡检

**完整性巡检**(每日凌晨执行):
* 输入:交易日历 + 全量股票池
* 逻辑:预期记录数 vs 实际记录数,缺口超过阈值告警
* 输出:每日完整性报告

**连续性巡检**(每日凌晨执行):
* 逻辑:对每只股票按日期排序,找断档
* 排除项:停牌期、退市后、上市前
* 输出:疑似缺失的 `(ts_code, date)` 列表

**停牌一致性巡检**(每日凌晨执行):
* 逻辑:`meta_suspension` 表中标记停牌的日期,日线表必须没有数据;反之亦然
* 输出:不一致清单

## Story E3-S3: 巡检结果落表 + 通知

* 所有发现的问题写入 `dq_findings` 表,带发现时间、规则、严重等级
* 严重问题(完整性 < 99%)立即推送(微信 / 邮件)
* 一般问题汇总到日报

## 验收标准

* 巡检任务每日固定时间执行,有执行日志
* 人为制造一个数据缺失,巡检能在下次执行时发现
* `dq_findings` 表可追溯每个问题的发现链路
