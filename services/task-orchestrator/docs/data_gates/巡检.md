# E3: 巡检模块独立部署(P0)

> 这是原文重点,但要明确边界:巡检只做"发现问题",修复交给 E6。

## Story E3-S1: 部署独立巡检节点

* **物理隔离**:不要和日常采集服务跑在同一台机器
* 推荐放在内网 NAS / 自有 PC / 性能稍好的边缘节点
* 调度时间避开交易时段和日常采集窗口(建议每日凌晨 2:00–5:00)

## Story E3-S2: 实现三类核心巡检

**完整性巡检**(每日凌晨执行):
* 输入:交易日历 + 全量股票池
* 逻辑:预期记录数 vs 实际记录数,缺口超过阈值告警
* 输出:每日完整性报告

**连续性巡检**(每日凌晨执行):
* 逻辑:对每只股票按日期排序,找断档
* 排除项:停牌期、退市后、上市前
* 输出:疑似缺失的 `(ts_code, date)` 列表

**停牌一致性巡检**(每日凌晨执行):
* 逻辑:`meta_suspension` 表中标记停牌的日期,日线表必须没有数据;反之亦然
* 输出:不一致清单

## Story E3-S3: 巡检结果落表 + 通知

* 所有发现的问题写入 `dq_findings` 表,带发现时间、规则、严重等级
* 严重问题(完整性 < 99%)立即推送(微信 / 邮件)
* 一般问题汇总到日报

## 验收标准

* 巡检任务每日固定时间执行,有执行日志 (已完成)
* 人为制造一个数据缺失,巡检能在下次执行时发现 (已验证)
* `dq_findings` 表可追溯每个问题的发现链路 (已实现)

## 技术细节说明 (Implementation Details)

### 1. 基础设施自愈 (Pre-flight Check)
为了确保在不稳定的内网环境中采集通路始终可用，巡检模块在执行前会进行自检：
* **探测逻辑**: 尝试通过 `HTTP_PROXY` 访问 `api.tushare.pro`。
* **自愈动作**: 若探测失败，通过 `docker.sock` 自动重启名为 `gost` 的容器。
* **告警**: 自愈成功或彻底失败均会触发邮件通知。

### 2. 邮件告警隧道方案 (SMTP over Proxy)
由于内网 Squid 代理 (`3128`) 禁用了 SMTP 465 端口，巡检模块采用了以下路由方案：
* **通道选择**: 显式将 SMTP 流量路由至国外代理通道 (`127.0.0.1:8118`)。
* **技术实现**: 使用 `PySocks` 对 `smtplib` 进行猴子补丁 (Monkey-patch)，建立 SMTPS 隧道。
* **安全保证**: 使用 `asyncio.Lock` 保护全局 Socket 替换，确保并发安全。

### 3. 数据审计算法
* **完整性**: 对比 `trade_cal` 与 `stock_basic_info`，排除停牌日期。
* **连续性**: 检查过去 5 个交易日的 K 线序列，寻找逻辑断档。
* **一致性**: 交叉验证 `stock_suspensions` 标记与 `stock_kline_daily` 实际落表情况。

### 4. 运维注意事项
* **运行模式**: 容器启动需设置环境变量 `APP_MODE=INSPECTOR`。
* **时区规范**: 强制要求 `Asia/Shanghai`，所有报告时间戳以此为准。
* **数据库**: 结果持久化至 `alwaysup.dq_findings` 和 `alwaysup.dq_reports`。
