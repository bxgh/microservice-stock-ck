# 盘后门禁 (Gate-3) 设计规范

**文档版本**: 3.0  
**最后更新**: 2026-05-04  
**状态**: 已优化 (数据重构 E1 阶段)

---

## 1. 核心原则

Gate-3 采用**分层审计**机制，区分“同步层”与“采集层”的不同质量标准。

**关键要求**：
1. **同步层 (K线/因子)**：Gate-3 仅负责审计 ClickHouse 与 MySQL 的数据镜像一致性。只要 CK 完整镜像了 MySQL，即视为合格。
2. **采集层 (分笔/快照)**：Gate-3 负责审计分笔数据相对于 K 线行情（真理来源）的缺失情况，并决策补采范围。
3. **架构解耦**：CK 侧的门禁不再主动判断市场总数缺口，不触发 `repair_kline` 等采集层任务，仅上报同步延迟或丢失。

## 3. 覆盖率与一致性审计规范

### 3.1 同步一致性 (K线与复权因子)

| 项目 | 说明 |
| :--- | :--- |
| **分子** | ClickHouse `stock_kline_daily` (或 `stock_adjust_factor`) 行数 |
| **分母** | **MySQL `stock_kline_daily` (或 `stock_adjust_factor`) 实时行数** |
| **公式** | `ClickHouse / MySQL × 100%` |
| **合格标准** | 必须 = 100% |
| **响应动作** | 仅上报同步失败告警，不干预数据源采集 |

### 3.2 数据完整性 (分笔 Tick 数据)

| 项目 | 说明 |
| :--- | :--- |
| **分子** | ClickHouse `tick_data` 当日去重股票数 |
| **分母** | **ClickHouse `stock_kline_daily` 当日去重股票数 (同步后的 CK 数据)** |
| **公式** | `Tick股票数 / K线股票数 × 100%` |
| **阈值** | < 95% 触发分级修复 |

> ⚠️ **禁止行为**: 严禁在 CK 侧使用 Redis 股票池或 Tushare 名单作为 K 线的审计分母。

## 4. 股票代码格式规范

| 场景 | 格式 | 示例 |
| :--- | :--- | :--- |
| K线/因子表 | 可能带前缀 | `sh600000`, `sz000001` |
| Tick表 | 纯代码 | `600000`, `000001` |
| 任务参数 | **必须纯代码** | `000001,000002,600001` |

**处理逻辑**: 从 K线表获取代码后，下发补采指令前需自动移除 `sh/sz` 前缀。

## 5. 分级修复策略 (仅针对分笔数据)

| 异常数量 | 策略 | 任务类型 | 参数 |
| :--- | :--- | :--- | :--- |
| **1-50** | 单点精准 | `stock_data_supplement` | `{"stocks": [...], "data_types": ["tick"]}` |
| **51-200** | 分片并行 | `stock_data_supplement` × 3 | 按 `xxhash % 3` 分组 |
| **> 200** | 分片定向 | `repair_tick` × 3 | 每个 Shard 携带该分片的 `failed_codes` |
