# 远程任务触发与参数化补采系统实现文档

## 1. 概述
为了实现移动端小程序对内网数据采集任务的控制，我们构建了一套“云端触发 -> 内网执行”的远程命令系统。该系统利用现有的 SSH 反向隧道基础设施，实现了安全的穿透控制，并支持带参数的任务执行（如按日期补采数据）。

## 2. 核心架构

### 2.1 数据流图 (Architecture 3.0 分布式)
```mermaid
graph TD
    User[小程序/管理员] -->|1. 插入命令| CloudDB[(云端 MySQL)]
    Orchestrator[Node 41 Orchestrator] -->|1.1 定时发射| CloudDB
    
    CloudDB -->|task_commands 表| LocalPoller41[Node 41 Poller (Shard 0)]
    CloudDB -->|task_commands 表| LocalPoller58[Node 58 Poller (Shard 1)]
    CloudDB -->|task_commands 表| LocalPoller111[Node 111 Poller (Shard 2)]
    
    LocalPoller41 -->|2. 执行| DP1[Worker 0]
    LocalPoller58 -->|2. 执行| DP2[Worker 1]
    LocalPoller111 -->|2. 执行| DP3[Worker 2]
    
    DP1 & DP2 & DP3 -->|3. 写入| ClickHouse[(ClickHouse 集群)]
```

### 2.2 关键组件
1.  **AlwaysUp Cloud MySQL (`task_commands`)**: 指令队列，支持 `FOR UPDATE SKIP LOCKED` 高并发竞争。
2.  **CommandEmitter (New)**: 运行在主节点 Node 41 的 `task-orchestrator` 中，负责在 T 日 18:00 自动计算并在 MySQL 中生成各分片的采集指令。
3.  **CommandPoller (Local)**: 部署在 41/58/111 上的 `shard-poller` 服务，根据 `shard_id` 认领其对应的指令。
4.  **Stale Task Sweeper**: 定期扫描 MySQL，回收执行超过 2 小时的“僵尸”任务。

## 3. 核心机制

### 3.1 目标日期计算规则 (6:00 AM Rule)
为了确保调度一致性，系统采用统一的 `target_date` 计算逻辑：
- **00:00 - 06:00**: 视为上一个交易日的数据，`target_date = Yesterday`。
- **06:00 - 23:59**: 视为当前交易日的数据，`target_date = Today`。

### 3.2 分片路由逻辑
Poller 通过 SQL 过滤参数中的 `shard_id` / `shard_index`：
- `shard-poller --shard 1`: 仅拉取 `shard_id=1` 的 PENDING 任务。
- `Node 41 (Orchestrator)`: 拉取 `shard_id=0` 或 `params` 中无分片信息的全局任务。

## 4. 数据库设计

### 云端表结构 (`alwaysup.task_commands`)
用于存储命令队列。

```sql
CREATE TABLE IF NOT EXISTS `task_commands` (
    `id` INT AUTO_INCREMENT PRIMARY KEY,
    `task_id` VARCHAR(64) NOT NULL COMMENT '对应 tasks.yml 中的任务ID',
    `params` JSON COMMENT '动态参数，如 {"date": "20260113"}',
    `status` ENUM('PENDING', 'RUNNING', 'DONE', 'FAILED') DEFAULT 'PENDING',
    `result` TEXT COMMENT '执行结果或错误日志',
    `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 4. 功能实现细节

### 4.1 CommandPoller (轮询器)
*   **位置**: `services/task-orchestrator/src/core/command_poller.py`
*   **逻辑**:
    1.  每 15 秒通过本地隧道 (`127.0.0.1:36301`) 连接云端 MySQL。
    2.  查询 `status='PENDING'` 的记录。
    3.  锁定记录（`SELECT ... FOR UPDATE SKIP LOCKED`）。
        *   **分布式安全**: 采用 `SKIP LOCKED` 机制，允许多个 Poller 实例（节点 41/58/111）并发工作而互不阻塞，自动跳过已被其他节点锁定的任务。
        *   更新状态为 `RUNNING`。
    4.  查找本地 `tasks.yml` 中对应的任务定义。
    5.  调用 `DockerExecutor` 执行任务，并阻塞等待完成。
    6.  更新执行结果 (`DONE`/`FAILED`) 和日志到云端。

### 4.2 支持的远程任务

目前系统支持以下专用补采任务，具体实现细节请参考各任务独立文档：

1.  **[分笔指定日期分片采集 (collect_tick_sharded)](remote_tasks/collect_tick_sharded.md)**
    *   用于分布式精确调度，无自动拆分逻辑。
    *   支持指定分片索引重跑。

2.  **[分笔数据按日补采 (repair_tick)](remote_tasks/repair_tick.md)**
    *   支持按日期、按股票列表补采。
    *   内置“智能分片”机制，应对大规模补采。

3.  **[K线数据按日补采 (repair_kline)](remote_tasks/repair_kline.md)**
    *   支持按日期触发智能自愈同步。
    *   自动联动盘后门禁审计。

4.  **[定向个股数据补充 (stock_data_supplement)](remote_tasks/stock_data_supplement.md)**
    *   支持多维度（Tick/K线/财务）个股深度体检与修复。

### 5.3 维护操作 (Maintenance)

由于 `task-orchestrator` 通常作为 Worker 节点的核心组件运行，当代码更新（如修复 Poller 逻辑）或依赖变更时，需要手动重构和重启。

**1. 重构镜像 (在项目根目录下执行)**
```bash
# 注意: 必须在根目录下运行，以包含 services/ 和 libs/ 上下文
docker build -t task-orchestrator:latest \
  -f services/task-orchestrator/Dockerfile . \
  --build-arg ENABLE_PROXY=true \
  --build-arg PROXY_URL=http://192.168.151.18:3128
```

**2. 启动或更新服务 (Worker 节点)**
```bash
# 适用于 Server 58 (需根据节点选择对应的 compose 文件)
docker compose -f services/task-orchestrator/docker-compose.poller-58.yml up -d

# 适用于 Server 111
# docker compose -f services/task-orchestrator/docker-compose.poller-111.yml up -d

# 查看日志确认启动成功
docker logs -f --tail 100 gsd-shard-poller
```
