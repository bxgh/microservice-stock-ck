## 📋 Story概述

### 目标
完成 EPIC-017 的最后闭环。在此前的研发中，我们已成功建设通过 GitHub 另类数据流式计算生态活跃信号 (HOT/EXTREME) 并注入选股大盘的业务链路。然而，**在主网实盘真金白银买入前，我们必须从统计学上向人类研究员证明：当某个开源仓库（例如 DeepSeek / vLLM）爆火之后，国内与之关联的 A 股映射板块确实具有短期或中期的超额收益 (Alpha)**。

### 验收标准
- [ ] 构建一个离线或一次性的历史回溯脚本，仿真拉取至少 3-6 个月以上的开源仓库变动记录谱系。
- [ ] 利用现有的 `EcoSignalStrategy` 将这批连续历史数据回放计算，生成对应的信号降临日期表。
- [ ] 获取对应时间段的大 A 股行情（提取相关领域的代表性成分股或者大盘涨跌幅）。
- [ ] 生成包含收益率透视与相关性的 Markdown 最终版报告。

---

## 🎯 需求分析

### 难点考量
1. **GitHub 数据回溯**: API 查询历史数据较为耗时，容易 Hit 到 Rate Limit，我们需要通过特定的 REST 参数获取某段时间的 Issue/PR 活动率，或者利用某些折中手段模拟这批历史数据进行管线测试。
2. **相关性对比**: 发出 EXTREME 信号后的 T+1，T+3，T+5 是否具备统计学意义的胜率？

---

## 🏗️ 技术设计

### 核心实现链路
1. **`tests/backtest/altdata_backtest.py`**:
   建立一个纯后台跑批的研究级脚本。它不参与每天的定时微服务，仅由研究人员在迭代因子时手动运用 `Quant-Strategy` 的模块执行。
2. **模拟回朔与真实验证**:
   由于真实的 GitHub 极深度的历史图谱（每一天的PR/Issue切片）获取难度极大，若真实验证难度过高，可以采用部分实盘与大量拟合模型进行回测走查。验证重点在于：**“我们的选股程序吃到这些分数的红利后，跑出来的模拟盘胜负关系如何”**。因此我们将直接调用现有 `CandidatePoolService` 等，在虚拟的前序时间打分。

### 替代/折中方案（如 API 受限）
若一次性提取巨量 GitHub 历史过载，将直接设计 `模拟数据驱动的回测器`：输入假设某年某月出现 HOT 信号，让大 A 股行情池回放以验证本量化管线的回溯功能是否稳固可行。
