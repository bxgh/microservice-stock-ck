# 分笔数据(Tick)采集与整理增强建议

## 1. 核心目标：构建"高保真"的数据底座

在不涉及复杂量化策略的前提下，数据采集与整理环节的核心任务是：**最大程度保留原始信息，同时提供高效的访问能力**。

---

## 2. 采集环节增强 (Data Acquisition)

### 2.1 盘口快照融合 (Quote Snapshot Fusion)
*   **现状**：通常 Tick 数据只包含成交信息（Price, Volume）。
*   **建议**：在采集 Tick 的同时，**必须**同步采集当时的盘口快照（买一/卖一至买五/卖五）。
    *   **原因**：没有盘口数据的 Tick 是"盲人摸象"。无法判断主动买卖方向，无法计算挂单压力。
    *   **实现**：如果数据源支持，将 `Tick` 和 `Snapshot` 合并存储，或通过时间戳精确对齐。

### 2.2 毫秒级时间戳对齐 (Millisecond Precision)
*   **建议**：确保采集的时间戳精确到毫秒（ms）。
    *   **原因**：高频交易中，1秒内可能发生几十笔交易。秒级时间戳会导致顺序混乱，无法还原真实的撮合顺序。

### 2.3 增量更新与断点续传 (Incremental Update)
*   **建议**：实现基于 `LastTickID` 或 `LastTimestamp` 的增量采集机制。
    *   **场景**：网络波动导致连接中断后，重连时能自动补全缺失的数据段，而不是重新下载全天数据。

---

## 3. 整理环节增强 (Data Organization)

### 3.1 统一数据模型 (Unified Data Model)
*   **建议**：定义一个标准的 Tick 数据结构，屏蔽不同数据源（新浪、腾讯、东财）的字段差异。
    *   **标准字段**：
        *   `timestamp`: 毫秒级时间戳
        *   `price`: 成交价
        *   `volume`: 成交量
        *   `turnover`: 成交额（部分源可能缺失，需计算）
        *   `direction`: 主动买卖方向（如源数据无此字段，标记为 Unknown，留给后续层推导）
        *   `bid1_price`, `ask1_price`: 买一/卖一价（关键！）
        *   `bid1_volume`, `ask1_volume`: 买一/卖一量

### 3.2 异常数据标记 (Data Quality Flagging)
*   **建议**：不要直接删除异常数据，而是添加一个 `quality_flag` 字段。
    *   `0`: 正常
    *   `1`: 价格跳变异常（超过阈值）
    *   `2`: 时间戳乱序
    *   `3`: 盘口数据缺失
    *   **原因**：有时候"异常"正是极端行情的真实反映。保留原始数据，打上标签，让下游分析决定是否使用。

### 3.3 高效存储分层 (Storage Tiering)
*   **热数据 (Hot)**：最近 7 天的数据。
    *   **格式**：Parquet / Feather。
    *   **特点**：读取极快，支持高频 I/O。
*   **冷数据 (Cold)**：历史归档数据。
    *   **格式**：Gzip 压缩的 CSV 或 Parquet。
    *   **特点**：压缩率高，节省成本。

### 3.4 交易日历对齐 (Trading Calendar Alignment)
*   **建议**：在整理数据时，严格对齐交易所的交易日历。
    *   **处理**：自动剔除非交易时段（如午休 11:30-13:00）的噪点数据，但保留集合竞价时段（09:15-09:25）。

---

## 4. 总结

在数据获取阶段，最重要的是**"不失真"**和**"可回溯"**。

1.  **采集**：尽可能多地抓取上下文信息（尤其是盘口数据）。
2.  **整理**：标准化格式，保留毫秒精度，打标签而非直接删除异常值。
3.  **存储**：使用 Parquet 等现代格式，为未来的高性能读取铺路。
