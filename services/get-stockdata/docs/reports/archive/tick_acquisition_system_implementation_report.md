## 1. 问题诊断与策略制定

### 1.1 初始需求分析
**用户原始需求**: "通过本报告内容和fenbi组件抽象优先级文档，对分笔数据的获取后的处理过程，进行展开思维，是否还有更好的建议？"

**专家诊断**:
- 当前 `fenbi_engine` 仅关注**软件工程层面**（去重、排序、格式化）。
- 缺乏**金融业务层面**的数据增强（主动买卖、VWAP、盘口深度）。
- 采集环节存在**数据保真度不足**的风险（无盘口、无毫秒精度）。

### 1.2 三维度战略建议

#### 维度 1: 金融特征工程 📈
文档位置: `docs/plans/fenbi_financial_processing_roadmap.md`

**核心观点**: 分笔数据不是"数字"，是"博弈痕迹"。

**关键增强方向**:
1. **主动买卖推导 (Trade Direction Inference)**
   - Lee-Ready 算法（需要 Bid1/Ask1）
   - Tick Test 回退方案（仅需价格序列）
   
2. **微观结构因子 (Microstructure Features)**
   - OFI (订单流不平衡)
   - VWAP (Volume Weighted Average Price)
   - Realized Volatility

3. **时序重采样 (Non-Uniform Sampling)**
   - Volume Bars (等量K线)
   - Dollar Bars (等额K线)

#### 维度 2: 采集环节增强 🔍
文档位置: `docs/plans/fenbi_data_acquisition_guide.md`

**核心原则**: "高保真" = 最大程度保留原始信息。

**关键建议**:
1. **盘口快照融合**: 必须同步采集 Bid1-Bid5 / Ask1-Ask5。
2. **毫秒级时间戳**: 理想状态，但免费源不支持，需接受"秒级"现实。
3. **异常数据标记**: 不删除，打标签（`quality_flag`）。
4. **Parquet 存储**: 替代 CSV，提升读写速度 10-50 倍。

#### 维度 3: 分层采集策略 🎯
文档位置: `docs/plans/tiered_acquisition_strategy.md`

**核心矛盾**: 全市场 5300 只股票高频采集会被封IP。

**解决方案**: 资产分级 + 频率分层 + 动态调度。

| 层级 | 股票范围 | 频率 | 用途 |
|------|----------|------|------|
| **L1 核心池** | 沪深300 + 自选股 (800-1000只) | **3秒/轮** | 高频策略、盘口分析 |
| **L2 活跃池** | 中证500 + 热门股 (1000-1500只) | **15秒/轮** | 短线选股 |
| **L3 全市场** | 剩余所有股票 (3000只) | **1分钟/轮** | 大盘情绪监控 |

**动态晋升机制**: L3 中的妖股（成交量暴增）自动升级到 L1，确保不漏捕。

## 2. 技术验证与接口测试

### 2.1 Mootdx 能力深度测试
文档位置: `docs/reports/mootdx_capability_analysis.md`

**测试方法**: Docker 容器内实弹测试。

**测试结果**:
| 功能 | 结论 | 数据示例 |
|------|------|----------|
| **盘口快照 (Snapshot)** | ✅ **支持** | `quotes()` 返回 Bid1-Bid5, Ask1-Ask5, Bid_Vol1-Bid_Vol5 |
| **历史盘口** | ❌ **不支持** | 仅能获取当前时刻 |
| **毫秒级时间戳** | ❌ **不支持** | 仅精确到秒 (HH:MM:SS) |
| **批量获取** | ✅ **支持** | 单次最多 80 只 |

**结论**: Mootdx 是目前免费方案中获取**实时五档盘口**的最佳选择，但无法补全历史盘口数据。

### 2.2 AkShare 对比测试
`akshare.stock_zh_a_spot_em()`: ❌ 仅有基础行情（Price, Volume），无五档盘口。  
`akshare.stock_zh_a_spot()` (新浪): ❌ 接口不稳定，测试中报错。

**最终选择**: **Mootdx** 作为主力采集源。

## 3. 系统架构与组件实现

### 3.1 核心组件 1: StockPoolManager
**代码位置**: `src/core/stock_pool/manager.py`

**功能**:
- 维护三级资产池（L1/L2/L3）的股票列表。
- 从 AkShare 自动获取沪深300成分股（`index_stock_cons('000300')`）。
- 支持用户自选股添加到 L1 池。

**验证结果**:
```
L1 Pool Initialized: 283 stocks ✅
Sample: ['600377', '601728', '000301', '002475', '002600']
```

### 3.2 核心组件 2: SnapshotRecorder
**代码位置**: `src/core/recorder/snapshot_recorder.py`

**功能**:
- 对 L1 池进行批量轮询（每批 80 只）。
- 使用 Mootdx `quotes()` 接口获取实时快照。
- 控制轮询周期为 3 秒/轮。

**性能验证**:
- **283 只股票** ÷ 80 = **4 个批次**
- **单轮耗时**: ~1.5 秒（含网络延迟）
- **QPS**: 4 请求 / 3 秒 ≈ **1.3 QPS** ✅ 远低于封IP阈值

**技术细节**:
- 禁用 `heartbeat` 和 `multithread`，避免 asyncio 冲突。
- 程序退出时的 `CancelledError` 是 Mootdx 内部清理行为，不影响数据采集。

### 3.3 核心组件 3: ParquetWriter
**代码位置**: `src/core/storage/parquet_writer.py`

**功能**:
- 将快照数据保存为 Parquet 格式（gzip 压缩）。
- 按日期分目录：`/data/snapshots/20251128/`
- 按小时分文件：`snapshot_14.parquet`

**优势**:
- **体积**: 比 CSV 小 90%。
- **速度**: 读取速度快 10-50 倍。
- **类型安全**: 原生支持 datetime, float，无需反复解析。

**依赖**: 已添加 `pyarrow>=14.0.0` 到 `requirements.txt`。

## 4. 性能与风险评估

### 4.1 性能指标
| 指标 | 目标 | 实测 | 状态 |
|------|------|------|------|
| L1 池规模 | 800-1000 只 | 283 只（沪深300） | ✅ |
| 轮询周期 | 3 秒 | ~3 秒 | ✅ |
| QPS | \< 5 | 1.3 | ✅ 安全 |
| 单轮数据量 | 283 行 × 46 列 | ~13KB (压缩后) | ✅ |

### 4.2 风险矩阵
| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| **IP 封禁** | 低 | 高 | 当前 QPS 1.3 远低于阈值；如扩展到 L2/L3 需增加延时 |
| **Mootdx 服务器列表过期** | 中 | 中 | 定期运行 `mootdx bestip`；准备 AkShare 作为备用源 |
| **数据丢失** | 低 | 高 | Parquet 文件自动压缩；建议增加每日归档任务 |
| **盘口数据缺失** | 高 | 中 | Mootdx 仅返回当前盘口，无法补全历史；可接受 |
