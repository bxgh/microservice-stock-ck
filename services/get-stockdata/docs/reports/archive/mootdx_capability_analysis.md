# Mootdx 接口能力深度测试报告

## 1. 测试结论

| 功能 | 结论 | 说明 |
|------|------|------|
| **盘口快照 (Snapshot)** | ✅ **支持 (仅实时)** | `quotes()` 接口可返回 Bid1-Bid5 / Ask1-Ask5 及其量。但无法获取历史时刻的快照。 |
| **毫秒级精度** | ❌ **不支持** | 无论是 `transactions()` 还是 `quotes()`，时间字段仅精确到秒 (HH:MM:SS) 甚至分。 |
| **历史分笔带盘口** | ❌ **不支持** | `transactions()` 仅包含成交价/量/方向，不含当时的买一/卖一。 |

## 2. 详细测试结果

### 2.1 盘口快照测试
使用 `Quotes.factory(market='std').quotes(symbol=['600000'])` 成功获取到浦发银行的实时快照。
**返回字段包含**:
- `bid1`, `bid_vol1` ... `bid5`, `bid_vol5`
- `ask1`, `ask_vol1` ... `ask5`, `ask_vol5`
- `price`, `volume`, `amount`

**数据示例**:
```
bid1: 11.48, bid_vol1: 1200
ask1: 11.49, ask_vol1: 3500
```

### 2.2 毫秒级精度测试
- `quotes()` 返回的 `servertime` 为 `15:00:00` 格式。
- `transactions()` 返回的 `time` 为 `14:59` 或 `15:00:03` 格式。
- **结论**: 无法满足高频交易对毫秒级时序的要求。

## 3. 架构建议

基于上述测试，针对"高保真采集"目标，建议采取以下策略：

### 3.1 实时采集 (Real-time)
- **双线程策略**:
    - **线程 A**: 循环调用 `quotes()` 获取快照 (建议 3秒/次)，存入 `Snapshot` 表。
    - **线程 B**: 循环调用 `transactions()` 获取最新分笔，存入 `Tick` 表。
- **数据融合**: 在内存中或入库后，基于时间戳 (秒级) 将 Tick 与最近的 Snapshot 关联。虽然有 1-3秒 的误差，但比完全没有盘口要好得多。

### 3.2 历史回溯 (Historical)
- **局限性**: 无法通过 Mootdx 补全历史数据的盘口信息。
- **解决方案**: 仅能依赖 Tick 数据的 `BuyOrSell` 字段（Mootdx 提供的 0/1/2）进行粗略的主动买卖判断，无法使用高精度的 Lee-Ready 算法。

## 4. 替代方案调研 (AkShare)
- `akshare.stock_zh_a_spot_em()`: 仅返回基础行情，无五档盘口。
- `akshare.stock_zh_a_spot()` (新浪): 接口不稳定，测试中报错。
- **结论**: 目前 Mootdx 仍是免费方案中获取实时五档盘口的最佳选择。
