# TODO 目录说明

本目录用于跟踪各个 Story 和 Epic 的待办事项。

---

## 📁 目录结构

```
docs/todo/
├── README.md              # 本文件
├── story_007_01.md        # Story 007.01 待办事项
├── story_007_02.md        # Story 007.02 待办事项（待创建）
└── ...                    # 其他 Story 待办事项
```

---

## 📋 使用规范

### 文件命名规则

- Story 级别：`story_XXX_YY.md`（如 `story_007_01.md`）
- Epic 级别：`epic_XXX.md`（如 `epic_007.md`）

### 内容规范

每个待办事项文档应包含：

1. **状态概览**：核心功能完成度
2. **待补充内容**：按优先级分类（P1/P2/P3）
3. **相关文档链接**：设计文档、验收报告等
4. **代码目录**：相关代码位置
5. **更新记录**：文档变更历史

### 优先级定义

- **P1 - 高优先级**：必须完成，影响功能稳定性
- **P2 - 中优先级**：重要但非紧急，影响用户体验
- **P3 - 低优先级**：优化项，按需完成

---

## 🔄 维护流程

### 创建待办文档

当一个 Story 核心功能完成验收后，创建对应的 todo 文档，记录：
- 已完成的内容
- 待补充的内容（测试、文档、优化等）

### 更新待办文档

- **完成事项**：在 `[ ]` 中打勾 `[x]`
- **新增事项**：添加到对应优先级分类下
- **调整优先级**：根据实际需求调整

### 关闭待办文档

当所有 P1 和 P2 事项完成后，在文档顶部标记 `状态: ✅ 已完成`。
