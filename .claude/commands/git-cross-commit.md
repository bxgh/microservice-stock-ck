# Cross-Domain Smart Commit - 跨域智能提交

## 命令描述

处理涉及多个域（前端、后端、基础设施）的复杂变更，提供全面的依赖分析、影响评估和协调提交策略，确保跨域变更的一致性和完整性。

## 执行流程

### 1. 全域变更扫描
- 执行 `git diff --name-only` 获取所有变更文件
- **域分类**: 将变更文件按域分类（前端/后端/基础设施）
- **跨域检测**: 识别是否存在跨域依赖关系
- **变更复杂度**: 评估跨域变更的复杂度和风险

### 2. 跨域依赖分析
- **前后端接口**: 分析API契约变更的影响
- **数据流映射**: 检查前后端数据模型变更
- **配置一致性**: 验证跨域配置的一致性
- **部署协调**: 分析跨域部署的依赖关系

### 3. 影响范围评估
- **服务影响**: 评估对各微服务的影响范围
- **用户体验**: 分析前端变更对用户体验的影响
- **系统稳定性**: 评估对整体系统稳定性的影响
- **回滚复杂度**: 评估跨域变更的回滚难度

### 4. 协调验证策略
- **分域验证**: 按域分别执行快速验证
- **集成测试**: 执行跨域集成测试
- **端到端测试**: 验证完整的用户场景
- **性能测试**: 评估跨域变更的性能影响

### 5. 协调提交
- **原子性提交**: 确保跨域变更的原子性
- **分阶段部署**: 提供安全的分阶段部署策略
- **回滚计划**: 制定详细的回滚应急方案
- **监控告警**: 配置变更后的监控和告警

## 使用方法

### 跨域提交
```
涉及前端和后端的跨域变更提交
```

### 复杂重构提交
```
我完成了用户认证系统的重构，涉及前端登录页面、后端API服务和数据库schema，请智能协调提交
```

### API集成提交
```
集成了新的第三方支付API，涉及前端支付页面、后端支付服务和配置文件，请智能分析并提交
```

## 跨域分析规则

### 域分类映射
```yaml
domain_mapping:
  frontend:
    - apps/*
    - packages/ui-components/*
    - packages/types/frontend/*

  backend:
    - services/*
    - packages/shared/*
    - packages/utils/*

  infrastructure:
    - infrastructure/*
    - scripts/*
    - docker-compose*.yml
    - .github/workflows/*
```

### 跨域依赖类型
1. **API契约变更**: 前后端接口定义变更
2. **数据模型变更**: 共享数据结构变更
3. **配置依赖变更**: 环境配置和服务配置变更
4. **部署依赖变更**: 服务部署顺序和依赖变更

### 风险评估矩阵
| 影响域数 | 变更类型 | 风险等级 | 建议策略 |
|---------|---------|---------|---------|
| 1 | 单域变更 | 低 | 标准提交流程 |
| 2 | 双域变更 | 中 | 协调提交，分步验证 |
| 3+ | 全域变更 | 高 | 完整测试，分阶段部署 |

## 特殊处理逻辑

### API契约同步检查
```bash
# 检查前后端API一致性
check_api_contract_sync() {
  frontend_api=$(extract_api_from_frontend)
  backend_api=$(extract_api_from_backend)

  if [ "$frontend_api" != "$backend_api" ]; then
    echo "⚠️  API契约不一致检测到"
    show_api_diff
  fi
}
```

### 数据模型一致性验证
```bash
# 验证共享类型定义
validate_shared_types() {
  frontend_types=$(extract_types_from_frontend)
  backend_types=$(extract_types_from_backend)

  compare_type_schemas "$frontend_types" "$backend_types"
}
```

### 部署依赖分析
```bash
# 分析服务部署依赖
analyze_deployment_dependencies() {
  services=$(get_changed_services)
  dependency_graph=$(build_service_dependency_graph $services)

  echo "📋 建议部署顺序:"
  topological_sort "$dependency_graph"
}
```

## 高级功能

### 自动冲突解决
- **接口版本冲突**: 自动生成兼容性适配层
- **配置覆盖冲突**: 智能合并配置文件
- **依赖版本冲突**: 推荐兼容的依赖版本

### 智能部署推荐
- **蓝绿部署**: 零停机部署策略
- **金丝雀发布**: 渐进式发布策略
- **回滚检查**: 自动回滚条件检测

## 注意事项

- **高复杂度**: 跨域变更是最复杂的提交类型
- **测试完整性**: 必须执行完整的测试套件
- **部署协调**: 需要仔细协调各域的部署顺序
- **回滚准备**: 必须准备完整的回滚方案

## 使用建议

### 推荐使用场景
- ✅ **API重构**: 前后端API接口重大调整
- ✅ **系统集成**: 新服务集成现有系统
- ✅ **架构升级**: 跨域架构和技术栈升级
- ✅ **功能完整实现**: 需要前后端配合的新功能

### 不推荐使用场景
- ❌ **简单Bug修复**: 单域内的简单问题修复
- ❌ **样式调整**: 纯前端样式变更
- ❌ **配置优化**: 纯基础设施配置调整

### 替代方案
- **分步提交**: 将跨域变更拆分为多个单域提交
- **特性分支**: 使用特性分支独立开发和测试
- **服务版本化**: 使用API版本控制避免冲突

这个命令专门处理复杂的跨域变更，确保多域协调的安全性和一致性。
