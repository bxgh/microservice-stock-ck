# Smart Incremental Commit - 智能增量代码提交

## 命令描述

基于Git diff快速扫描和提交前端代码变更，**仅分析Git diff输出的变更文件**，避免全项目扫描，实现秒级精准提交。

## 执行流程

### 1. Git Diff 快速扫描
- 执行 `git diff --name-only` 获取变更文件列表
- **仅处理变更文件**，不扫描整个项目
- 快速分析变更类型：新增/修改/删除文件
- 基于文件路径判断影响范围：组件/页面/API/配置等

### 2. 变更文件内容分析
- **语法检查**: 仅检查变更文件的语法错误
- **规范检查**: 快速验证变更代码的基本规范
- **类型检查**: 检查变更部分的类型安全性
- **变更摘要**: 生成变更内容的简要描述

### 3. 最小化关联分析
- **仅检查同级文件**: 分析同目录下的相关文件
- **直接依赖检查**: 仅检查明显的import/export关系
- **跳过深度扫描**: 不进行全项目依赖分析
- **基础关联**: 基于文件名和路径的简单关联

### 4. 快速验证 (可选)
- **基础语法验证**: 确保变更文件可以正常解析
- **导入验证**: 检查新增的导入是否存在
- **类型检查**: 验证TypeScript类型定义
- **跳过运行时测试**: 不启动服务进行动态验证

### 5. 增量注释和文档
- **代码注释**: 为新增/修改的函数添加中文注释
- **组件文档**: 更新修改组件的Props和Events文档
- **API文档**: 更新变更API的接口说明
- **变更日志**: 记录本次变更的技术细节

### 6. 精准提交
- **暂存变更**: 只添加实际修改的文件
- **智能Commit**: 基于变更内容生成中文提交信息
- **影响范围**: 在commit信息中明确说明影响范围
- **关联提交**: 如需要，自动提交相关的配置文件

## Git Diff 快速扫描规则

### 变更文件获取
```bash
# 仅获取变更文件，避免全项目扫描
git diff --name-only --cached HEAD~1  # 已暂存的变更
git diff --name-only HEAD             # 工作区变更
```

### 快速变更分析
- **基于文件路径**: 根据文件路径快速判断文件类型
- **基于扩展名**: .vue(组件)、.ts(逻辑)、.scss(样式)、.json(配置)
- **基于目录结构**: /components/、/views/、/api/、/utils/
- **基于变更大小**: 统计增删行数，避免处理超大型变更

### 最小化影响评估
- **同级文件检查**: 仅检查变更文件所在目录的相关文件
- **直接导入分析**: 仅分析文件中的直接import语句
- **跳过递归分析**: 不进行深度的依赖链分析
- **快速类型推断**: 基于文件扩展名和内容片段推断类型

## 性能优化策略

### 扫描范围限制
- **仅处理Git diff输出**: 从不扫描git diff之外的文件
- **文件数量限制**: 超过50个变更文件时提示分批处理
- **文件大小限制**: 单个文件超过1000行变更时跳过详细分析
- **超时保护**: 单次分析设置30秒超时限制

### 快速验证模式
- **语法优先**: 仅检查明显的语法错误
- **类型简化**: 不进行复杂的类型推导
- **规范放宽**: 仅检查关键的代码规范问题
- **跳过测试**: 不进行任何运行时验证

### 内存和性能优化
- **流式处理**: 逐个处理变更文件，避免全量加载
- **缓存利用**: 复用已分析的文件信息
- **并行处理**: 对独立的变更文件进行并行分析
- **早退机制**: 发现严重错误时立即停止分析

### 增量注释规则
- **新增函数**: 为新函数添加完整中文注释
- **修改函数**: 更新修改函数的注释
- **组件Props**: 为新增Props添加类型和说明
- **变更说明**: 在注释中说明变更原因

## Commit Message生成规则

### 类型识别 (基于变更内容)
- **feat**: 新增组件、页面、功能
- **fix**: 修复Bug、错误处理、兼容性
- **refactor**: 代码重构、性能优化
- **docs**: 注释、文档更新
- **style**: 样式调整、代码格式
- **test**: 测试相关文件
- **chore**: 配置、工具、构建文件

### 影响范围 (基于变更文件)
- **components**: 组件库相关变更
- **dashboard**: 监控仪表板相关
- **api**: API集成和服务相关
- **config**: 配置文件相关
- **pages**: 页面路由和布局相关

### 中文描述格式
```
类型(范围): 简洁描述变更内容

## 变更详情
- 主要变更点1
- 主要变更点2
- 影响范围说明

🤖 Generated with [Claude Code](https://claude.com/claude-code)

Co-Authored-By: Claude <noreply@anthropic.com>
```

## 质量标准

### 增量检查通过标准
- 变更文件无语法错误
- 变更逻辑正确无误
- 关联功能正常工作
- 新增注释规范完整
- 提交信息准确描述变更

### 检查失败处理

- 逻辑错误 → 指出问题位置
- 功能问题 → 提供调试方向
- 规范问题 → 给出修改建议

## 注意事项

- **无变更检查**: 如果没有Git变更，立即提示用户
- **大型变更**: 超过50个文件变更时自动分批处理
- **超大文件**: 单个文件变更超过1000行时跳过详细分析
- **冲突处理**: 如有Git冲突，先解决冲突再执行
- **性能优先**: 优先保证执行速度，跳过耗时的深度分析
- **准确性权衡**: 快速模式下可能会遗漏一些间接影响

## 使用建议

### 推荐使用场景
- ✅ **日常开发提交**: 频繁的小范围代码变更
- ✅ **CI/CD集成**: 自动化的代码提交流程
- ✅ **大型项目**: 文件数量庞大的前端项目
- ✅ **快速迭代**: 需要快速提交验证的开发模式

### 不推荐使用场景
- ❌ **重大重构**: 需要深度分析影响范围的架构变更
- ❌ **复杂依赖**: 涉及复杂依赖关系的模块变更
- ❌ **全面测试**: 需要完整回归测试的重要功能
- ❌ **首次提交**: 空项目的首次代码提交

### 替代方案
- **深度分析**: 使用 `--deep` 参数启用完整分析模式
- **手动提交**: 对于复杂变更，使用传统git命令手动处理
- **分批提交**: 将大型变更拆分为多个小型提交
