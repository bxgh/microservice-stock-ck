# 架构设计准则：分布式存储的防御性选型
版本：1.0 | 状态：正式发布

## 1. 核心原则：防御过度设计 (Anti-Overengineering)

在处理 ClickHouse 集群（如本项目中的 41, 58, 111 架构）时，严禁盲目将所有表结构“集群化”。错误的分布式选型会带来元数据同步开销、IO 放大以及不必要的网络排队。

## 2. 选型判定矩阵

| 数据特征 | 推荐架构 | 理由 |
| :--- | :--- | :--- |
| **全量 Tick/快照** (日增 >1亿行) | `Local` + `Distributed` | 必须利用多机磁盘 IO 和 CPU 并行处理。 |
| **K线/逐笔成交** (日增 >100万行) | `Local` + `Distributed` | 考虑长期存储压力，需要水平分片。 |
| **复盘/财务/基础资料** (日增 <1000行) | **单机 MergeTree (Node 41)** | 数据量极小，单机聚合效率远高于集群 Scatter-Gather。 |
| **策略信号/审计日志** (日增 <10万行) | **单机 MergeTree (Node 41)** | 减少分布式元数据竞争，保证写入稳定性。 |

## 3. 强制约束 (Hard Rules)

1.  **数据量门槛**：单表预计总数据量**小于 100 万行**或物理体积**小于 100MB** 时，禁止创建集群分布式表。
2.  **单一事实来源**：对于结论性数据（如复盘分析），必须指定唯一物理存储节点（默认为 Node 41），禁止在其他工作节点（58, 111）创建物理副本。
3.  **元数据隔离**：工作节点（58, 111）的元数据（Tables/Views）应保持极简。不参与计算或展示的业务结论表，不应在工作节点可见。
4.  **按需解耦，逻辑合并**：
    *   物理层：根据数据来源（资金类型、策略类型）进行垂直分表，提高更新灵活性。
    *   逻辑层：通过 `VIEW`（视图）在主控节点进行 Join，提供统一接口。
5.  **删除前审计（Drop Pre-flight）**：任何 `DROP TABLE ... ON CLUSTER` 指令执行前，必须先执行 `SELECT name, total_rows FROM system.tables WHERE name = '目标表名'`，确认目标表为本次新建且无存量数据（`total_rows = 0`）后，方可执行删除。**严格禁止对 `total_rows > 0` 的生产表执行集群级删除**。

## 4. 典型反面案例

*   **案例**：为年增 250 行的“每日复盘表”创建 3 节点分布式分片。
*   **后果**：导致查询时 3 个线程同时启动、3 次网络握手、3 次 IO 读取，仅为了检索几行文本数据。这不仅浪费了硬件资源，还增加了集群 `Zookeeper/Keeper` 的元数据同步负担。
