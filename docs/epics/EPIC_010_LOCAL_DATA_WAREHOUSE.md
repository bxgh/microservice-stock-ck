## 1. 问题陈述

### 当前挑战
1. **数据孤岛**: 各微服务 (`get-stockdata`, `quant-strategy`) 各自维护数据连接，无法共享
2. **无本地持久化**: 财务数据按需从腾讯云 MySQL 获取，导致重复 API 调用
3. **云端依赖**: 当腾讯云 MySQL 响应慢或不可用时，关键策略被阻塞
4. **因子存储缺失**: 无法存储计算后的因子值，无法支撑回测

### 业务影响
- 财务数据查询延迟 5+ 秒
- 访问云端 MySQL 产生冗余网络流量
- 无历史因子数据，回测功能受限

---

## 3. 范围定义

### 本期范围 (Phase 1)
- ClickHouse 作为主要时序存储
- Redis 作为热数据缓存层
- 腾讯云 MySQL → 本地 ClickHouse 同步管道
- 统一数据访问 API (新建 `data-warehouse` 服务)
- 因子值存储与查询

### 后续规划 (Phase 2)
- 实时流式摄入 (Kafka)
- 机器学习特征仓库
- 多租户数据隔离

## 4. 数据分类

| 数据类别 | 真实数据源 | 采集方式 | 本地存储 | 更新策略 |
|----------|------------|----------|----------|----------|
| 分笔数据 (Tick) | Mootdx | `data-collector` 实时采集 | ClickHouse | 实时写入 |
| 日K线 (OHLCV) | Baostock | `data-collector` 定时采集 | ClickHouse | 每日 T+1 |
| 财务报表 | AkShare | `data-collector` 定时采集 | ClickHouse | 季度更新 |
| 估值指标 | AkShare | `data-collector` 定时采集 | ClickHouse | 每日更新 |
| 因子值 | 本地计算 | `quant-strategy` 生成 | ClickHouse | 按需计算 |
| 实时行情 | Mootdx | `data-collector` 实时采集 | Redis (TTL 3秒) | 实时更新 |
| 元数据 | Baostock | `data-collector` 定时采集 | PostgreSQL | 每周同步 |

> **说明**: 腾讯云 MySQL 作为**云端备份**存储，接收 `data-collector` 的双写数据，用于灾备和历史归档。

## 5. 用户故事

### Story 10.0: data-collector 服务脚手架 (新增)
**优先级**: P0  
创建 `data-collector` 微服务，实现基础采集框架和调度器。

### Story 10.1: ClickHouse 表结构设计
**优先级**: P0  
设计并创建核心表: `kline_daily`, `financials`, `valuation_history`, `factor_values`。

### Story 10.2: 日K线采集任务
**优先级**: P0  
实现从 Baostock 采集日K线数据，双写到 ClickHouse 和腾讯云 MySQL。

### Story 10.3: 财务数据采集任务
**优先级**: P0  
实现从 AkShare 采集财务报表数据，按季度更新。

### Story 10.4: data-warehouse 服务脚手架
**优先级**: P1  
创建 `data-warehouse` 微服务，提供统一数据访问 API。

### Story 10.5: 统一数据访问 API
**优先级**: P1  
实现 K线、财务、因子等数据的查询接口。

### Story 10.6: 因子存储集成
**优先级**: P1  
将因子库与数据仓库集成，实现因子值持久化。

### Story 10.7: 实时行情采集
**优先级**: P2  
实现从 Mootdx 采集实时行情，写入 Redis 缓存。
