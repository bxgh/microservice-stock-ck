## DL-2026-01-07: 分笔数据采集 SBF 顺序批回溯策略

| 项 | 内容 |
|-----|------|
| **问题** | mootdx 分笔数据中 09:25 集合竞价位置不固定，传统搜索矩阵 (Search Matrix) 覆盖率仅 ~75% 且不准确 |
| **决策** | 采用 SBF (Sequential Batch Fetching) 策略，从 15:00 开始逐批向前回溯直到发现 09:25 |
| **备选方案** | 1) 固定大偏移量索引 (在高活跃股会失效) 2) 二分查找法 (因 API 分批且不稳定而难以实现) |
| **原因** | 1) 100% 覆盖潜力 (只要源端有) 2) 自适应不同活跃股 3) 采集过程通过小批次保证稳定性 |
| **后果** | 采集全市场耗时增加 (约 2 小时)；需要更细致的连接清理逻辑 |
| **相关** | [WALKTHROUGH_SBF_TICK_20260107.md](../reports/WALKTHROUGH_SBF_TICK_20260107.md) |

## DL-2026-01-04: ClickHouse Active-Active 复制

| 项 | 内容 |
|-----|------|
| **问题** | 单节点 ClickHouse 存在单点故障风险 |
| **决策** | 采用 Active-Active 复制 (ReplicatedReplacingMergeTree) |
| **备选方案** | 1) 主从复制 2) 外部 ZooKeeper 集群 |
| **原因** | 1) 使用内置 Keeper 无需外部依赖 2) 支持双写 3) 配置简单 |
| **后果** | 需迁移现有表到 Replicated 引擎；2 节点 Keeper 有脑裂风险 |
| **相关** | [PROGRESS_REPORT_20260104.md](../reports/PROGRESS_REPORT_20260104.md) |

## DL-2025-12-XX: 数据源路由策略

| 项 | 内容 |
|-----|------|
| **问题** | 本地无法直接访问外网数据源 |
| **决策** | 采用多级数据源路由：本地 mootdx → 云端 Baostock/AkShare |
| **架构** | `mootdx-source` 作为路由层，根据数据类型选择源 |
| **原因** | 1) 实时数据用 mootdx 低延迟 2) 历史数据用云端 API 稳定 |
| **后果** | 需维护 SSH 隧道和代理配置 |

## DL-2025-12-XX: task-scheduler → task-orchestrator 重构

| 项 | 内容 |
|-----|------|
| **问题** | 原 task-scheduler 功能单一，难以扩展 |
| **决策** | 重构为 task-orchestrator，支持动态任务注册 |
| **变化** | 1) 配置驱动 (tasks.yml) 2) 支持 Workflow 3) 容器化执行 |
| **原因** | 提高灵活性，支持复杂 DAG 任务依赖 |
| **后果** | task-scheduler 废弃 |

## DL-2025-XX-XX: 时区统一使用 Asia/Shanghai

| 项 | 内容 |
|-----|------|
| **问题** | A 股交易时间有严格时区要求 |
| **决策** | 全系统统一使用 `Asia/Shanghai` (CST) 时区 |
| **范围** | 代码、调度、日志、数据库时间戳 |
| **原因** | 避免时区转换错误导致的数据问题 |
| **后果** | 所有时间操作必须显式指定时区 |
