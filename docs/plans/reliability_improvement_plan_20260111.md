# 数据采集可靠性与稳定性改进计划

**日期**: 2026-01-11
**背景**: 在单节点基准测试中，发现上游 TDX 服务器存在“假死”现象（连接成功但返回空数据），导致采集器误判为正常结束，未触发重试。同时，静态 IP 列表无法应对网络环境的动态变化。

## 1. 核心问题分析

1.  **"假死" IP 欺骗**: 部分 TDX 服务器对请求返回 HTTP 200 OK，但响应体为空 ("No Data")。现有逻辑将其视为“无成交”或“停牌”，导致数据缺失。
2.  **被动重试局限**: 目前的重试机制仅针对网络异常（超时、连接断开）。对于业务层的“空数据”异常缺乏判断力。
3.  **静态池老化**: 硬编码的 `TDX_HOSTS` 会随着时间推移失效，缺乏自动补充和剔除机制。

## 2. 改进方案

### 2.1 增强数据校验 (Smart Validation)
**目标**: 识别“假死”响应，强制触发轮询重试。

*   **策略**:
    *   **基准对比**: 引入“已知活跃股票”（如平安银行 000001, 贵州茅台 600519）。如果在交易日这些股票返回空数据，直接判定为当前链路故障。
    *   **交易日历检查**: 结合交易日历，如果是交易时段且由“无数据”返回，抛出自定义异常 `DataEmptyError` 而非静默返回。
    *   **效果**: 抛出异常后，worker 的重试逻辑(`tenacity`)会自动捕获，并由 `mootdx-api` 的轮询机制由下一个 IP 接手重试。

### 2.2 主动健康检查与熔断 (Active Health Check & Circuit Breaker)
**目标**: 在 `mootdx-api` 服务端，将坏节点挡在门外。

*   **实施**:
    *   **后台巡检任务**: `mootdx-api` 启动一个后台线程，每分钟轮询一遍连接池中的 IP。
    *   **探针请求**: 向每个 IP 发送一个极轻量的标准请求（例如查询上证指数 `999999` 的最新分笔）。
    *   **熔断机制**:
        *   如果 连续 N 次（如 3 次）探针失败或返回空：将该 IP 标记为 `UNHEALTHY`，暂时移出可用列表。
        *   **冷却恢复**: M 分钟后（如 30 分钟），尝试发送一次探针，如果成功则恢复为 `HEALTHY`。

### 2.3 动态 IP 扫描与补充 (Dynamic Discovery)
**目标**: 解决 IP 池枯竭问题，建立高质量的后备资源池。

*   **实施策略**:
    *   **优先扫描 (Priority Scanning)**:
        *   维护 `PRIORITY_CIDR` 列表，优先覆盖高潜力的云厂商和券商核心网段。
        *   **重点网段**: 华为云段 (如 `124.71.x.x`), 深圳电信/海通核心段 (如 `119.147.x.x`)。这些网段通常部署有“原厂级”行情转发节点，质量极高。
    *   **扫描风控 (Risk Control)**:
        *   **随机延迟**: 在执行端口 (7709) 扫描时，强制引入 0.1s~0.5s 的随机延迟 (Jitter)，避免触发云厂商防火墙的 "Port Scan" 拦截规则。
        *   **低频作业**: 仅在非交易时段（如凌晨 02:00）执行全量扫描。
    *   **验证入库**: 对扫描到的 IP 进行高强度测压（模拟 4 并发），仅将 **响应时间 < 200ms** 且 **数据完整** 的 IP 存入 Redis `TDX_CANDIDATES`。
    *   **动态加载**: `mootdx-api` 监听 Redis 变更，支持无需重启的热加载。

## 3. 实施路线图

| 阶段 | 任务 | 优先级 | 预计工时 |
| :--- | :--- | :--- | :--- |
| **阶段一** | **数据校验增强** (Worker 端修改) | **P0 (高)** | 2 天 |
| **阶段二** | **连接池熔断机制** (API 端修改) | **P1 (中)** | 3 天 |
| **阶段三** | 动态扫描与热加载 | P2 (低) | 待定 |
