# 数据安全操作规范

## 核心原则

**任何可能修改生产数据或代码的操作，必须先获得用户明确批准。**

---

## 禁止的自动操作

以下操作**严格禁止**在未经用户明确授权的情况下执行：

### 数据库操作
- ❌ `DELETE` / `DROP` / `TRUNCATE` / `ALTER TABLE ... DELETE`
- ❌ `UPDATE` / `INSERT` / `REPLACE`
- ❌ 任何修改数据的 SQL 语句
- ❌ 数据同步/迁移脚本

### 代码修改
- ❌ 修改生产环境配置文件
- ❌ 修改默认参数值（尤其是数据采集相关）
- ❌ 修改数据库连接字符串
- ❌ 将代码文件复制到运行中的容器

### 系统操作
- ❌ 重启服务
- ❌ 清理缓存/日志（除非明确是临时测试数据）
- ❌ 修改环境变量

---

## 必须遵循的流程

### 1. 只读查询（无需授权）
```sql
✅ SELECT ... FROM ...
✅ SHOW TABLES / DESCRIBE TABLE
✅ EXPLAIN QUERY
```

### 2. 分析和报告（无需授权）
- ✅ 生成对比报告
- ✅ 分析问题根因
- ✅ 提出修复方案（但不执行）

### 3. 修改操作（必须授权）

**标准流程**：
```
第一步: 描述要做什么
  - 明确说明影响范围
  - 列出将要修改的数据/代码
  - 说明可能的风险

第二步: 等待用户明确批准
  - 用户回复 "批准" / "执行" / "继续"
  - 或用户明确拒绝

第三步: 执行并报告结果
  - 严格按照批准的范围执行
  - 报告执行结果
```

---

## 关键词识别

### 触发"只读模式"的关键词
- "检查" / "查看" / "分析" / "对比"
- "验证" / "测试" / "评估"
- "生成报告" / "给我看看"

### 触发"需要授权"的关键词
- "修改" / "删除" / "更新" / "同步"
- "修复" / "清理" / "重置"
- "执行" / "运行" / "部署"

**重要**: 即使用户使用了"执行"类关键词，如果涉及生产数据，仍需二次确认。

---

## 特殊场景

### 测试环境 vs 生产环境
- **测试环境**: 可以相对自由操作（但仍需说明）
- **生产环境**: 严格遵循授权流程
- **不确定时**: 按生产环境处理

### 单条数据 vs 批量数据
- **单条测试数据**: 可以在说明后操作
- **批量数据**: 必须获得授权
- **不确定影响范围**: 必须获得授权

### 可逆 vs 不可逆操作
- **可逆操作**（如添加数据）: 说明后可执行
- **不可逆操作**（如删除）: 必须获得授权

## 违规案例（禁止重复）

### 案例 1: 未授权删除数据
```sql
-- ❌ 错误示范
ALTER TABLE stock_kline_daily DELETE WHERE stock_code = 'sh.600519'
```

**正确做法**：
```
AI: "我发现数据有问题，建议删除 sh.600519 从 2025-12-01 开始的数据。
     影响范围: 约 20 条记录
     是否批准执行？"
用户: "批准"
AI: [执行删除操作]
```

### 案例 2: 未授权修改代码
```python
# ❌ 错误示范
adjustflag: str = "3"  # 直接修改默认值
```

**正确做法**：
```
AI: "发现 data-collector 默认采集前复权数据，建议修改为采集原始数据。
     需要修改文件: datasource.py
     影响: 未来所有新采集的数据将是原始价格
     是否批准修改？"
用户: "批准"
AI: [修改代码]
```

## 紧急情况处理

即使在紧急情况下（如发现严重 bug），也必须：
1. 立即报告问题
2. 说明紧急程度
3. 提供临时缓解方案
4. **等待用户决策**

❌ 不要因为"紧急"就跳过授权流程

---

## 检查清单

在执行任何操作前，问自己：
- [ ] 这个操作会修改数据吗？
- [ ] 这个操作会修改代码吗？
- [ ] 这个操作是可逆的吗？
- [ ] 用户明确授权了吗？
- [ ] 我已经说明了影响范围吗？

**如果任何一项答案是"否"或"不确定"，停止并请求用户指示。**
