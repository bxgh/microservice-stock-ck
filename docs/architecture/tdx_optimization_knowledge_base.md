# TDX 连通性与连接池优化技术知识库

本库汇总了在 TDX 行情采集系统优化过程中的核心技术发现与实操规范。

## 1. 网络连通性深度实证
### 1.1 物理出口直连封禁 (RST Interception)
- **发现**: 绝大多数 TDX 行情节点对非 IDC/授权网段的物理 IP 存在精准端口封禁（7709, 7727 等）。
- **表现**: TCP 握手阶段立即收到远程内核发送的 `RST` 报文 (`Connection refused`)。
- **规避**: 必须通过具备授权出口的代理隧道（如 Dove/GOST）进行穿透。

### 1.2 代理带宽与并发冗余
- **结论**: 单路 GOST/Dove 隧道在 TCP 握手层面具有极高冗余（100+ 并发下 P95 握手延迟 < 30ms）。
- **瓶颈点**: 真正的瓶颈在于 **远程 TDX 节点的业务处理质量** 及其对单源 IP 的频控（Rate Limiting）。

## 2. 多网卡负载均衡实现逻辑
### 2.1 源 IP 绑定 (Socket Binding)
- **挑战**: 仅在代码层执行 `bind((LOCAL_IP, 0))` 并不能保证流量从对应的网卡流出，流量通常会根据 Metric 最小的缺省网关路由。
- **解决方案**: **策略路由 (PBR)**。必须为每个本地 IP 建立独立的路由表，并配置 `ip rule` 强制报文从对应的物理接口流出。

### 2.2 容器环境下的 Monkey Patch
- **实战**: 在不修改第三方库的前提下，通过修改 `socket.socket` 基类，在 `connect` 前强制注入 `bind` 逻辑，是实现多出口分流的最高效手段。

## 3. 高可靠连接池管理
### 3.1 幽灵节点 (Ghost Nodes) 过滤
- **陷阱**: 部分节点虽然 TCP 握手能秒回成功，但业务层处于静默状态，无法通过行情查询。
- **对策**: **协议级深度拨测**。探活逻辑必须包含一次真实的业务请求（如 `get_security_count`）。只有数据握手成功的节点才计入有效池。

### 3.2 动态评分与热更新机制
- **最优配方**: 将连接均匀分散在 **15+ 个高评分节点** 上，每个节点通过不同的 **物理网卡** 维持 3-4 个长连接。
- **价值**: 此方案在不增加系统拓扑复杂度（单代理架构）的前提下，通过“优选快连”能使 P99 数据延迟下降 60% 以上。

## 4. 故障排查 checklist
- [ ] **RST 报错**: 检查是否旁路规则失效导致流量走到了物理出口。
- [ ] **Timeout 报错**: 检查代理服务（GOST）是否宕机或 Dove 链路是否拥塞。
- [ ] **数据为空**: 确认该节点是否为“幽灵节点”，执行 `discovery.py` 进行全量重新评分。
