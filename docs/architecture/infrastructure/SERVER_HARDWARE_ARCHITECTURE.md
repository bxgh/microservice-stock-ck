## 🌐 Server 41 网络专项架构 (三网卡全隔离)

为了实现全集群一致的系统稳定性，Server 41 同步采用三网卡物理隔离架构，将管理、同步和采集流量彻底分流。

### 1. 物理网卡部署

| 接口名称 | 物理 MAC | 绑定 IP | 用途 | 优先级 (Metric) |
|:---------|:---------|:--------|:-----|:---------------|
| **ens32** | `00:50:56:b7:4b:22` | `192.168.151.41` | **内部集群同步流** (CH Sync, Keeper Raft) | 300 |
| **ens34** | `00:50:56:b7:7c:81` | `192.168.151.47` | **HTTP 采集代理流** (Squid 3128) | 200 |
| **ens35** | `00:50:56:b7:4c:b9` | `192.168.151.49` | **SSH 隧道加密流** (Gost 8118 / Dove) | 100 |

### 2. 路由分流策略

系统通过 `Netplan` 策略路由实现了物理层的流量锁定：

- **集群心跳锁定 (NIC 1)**：
  - 去往 `192.168.151.58` 和 `192.168.151.111` 的流量强制绑定到 `ens32`。
- **HTTP 代理锁定 (NIC 2)**：
  - 去往代理网关 `192.168.151.18` 的流量强制绑定到 `ens34`。
- **默认出口与加密隧道 (NIC 3)**：
  - 默认网关优先级最高设在 `ens35` (Metric 100)，确保 SSH 隧道和 Gost 数据包优先使用独立物理链路。

### 3. 配置持久化路径
- **Netplan 配置文件**: `/etc/netplan/60-triple-nic-config.yaml`
- **生效日期**: 2026-01-08

## 📈 运维建议
- **监控指标**: 需重点关注两个网卡的流量吞吐率 (Transmit/Receive Bytes)，确保高负载期间 `ens32` 的延迟保持在 < 1ms。
- **扩展性**: 若 Server 41 或 58 在后续高并发写入中出现类似 `Code: 319` 报错，建议同步扩展为双网卡架构。
