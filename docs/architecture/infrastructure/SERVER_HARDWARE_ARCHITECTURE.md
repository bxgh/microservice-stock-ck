## 🌐 Server 111 网络专项架构 (双网卡分流)

为了解决大规模 Tick 数据采集时导致的集群心跳延迟（Session moved 错误），Server 111 采用了双网卡物理隔离架构。

### 1. 物理网卡部署

| 接口名称 | 物理 MAC | 绑定 IP | 用途 | 优先级 (Metric) |
|:---------|:---------|:--------|:-----|:---------------|
| **ens32** | `00:50:56:b7:5f:5b` | `192.168.151.111` | **内部集群通讯流** (CH Sync, Keeper Heartbeat) | 200 |
| **ens34** | `00:50:56:b7:06:6c` | `192.168.151.36` | **外部数据采集流** (Proxy, External APIs) | 100 |

### 2. 路由分流策略

系统通过 `netplan` 实施了精细化路由控制，确保内部集群同步不因采集流量而产生抖动：

- **L2/L3 静态路由**：
  - 访问 `192.168.151.41` (S41) 强制绑定到 `ens32`。
  - 访问 `192.168.151.58` (S58) 强制绑定到 `ens32`。
- **默认出口 (Default Gateway)**：
  - 设置 `ens34` 的 Metric 为 `100`，`ens32` 为 `200`。
  - 所有非特定路由流量（即外网采集流量）优先从新网卡 `ens34` 出发。

### 3. 配置持久化路径
- **Netplan 配置文件**: `/etc/netplan/60-ens34-config.yaml`
- **生效日期**: 2026-01-08

## 📈 运维建议
- **监控指标**: 需重点关注两个网卡的流量吞吐率 (Transmit/Receive Bytes)，确保高负载期间 `ens32` 的延迟保持在 < 1ms。
- **扩展性**: 若 Server 41 或 58 在后续高并发写入中出现类似 `Code: 319` 报错，建议同步扩展为双网卡架构。
