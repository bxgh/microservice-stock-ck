# 个股分笔数据全维度校验方案 (Level 1-3)

本方案定义了**单只股票**在盘后进行全维度数据质量校验的标准与逻辑。该逻辑被用于 `TickValidator` 中，作为数据自愈的核心判断依据。

## 1. 校验金字塔 (Validation Pyramid)

对任意一只股票（Stock Code），必须依次通过以下三层校验，方可认定为"合格"。

| 层级 | 名称 | 关注点 | 校验逻辑 (伪代码) | 失败结论 |
| :--- | :--- | :--- | :--- | :--- |
| **L1** | **存在性**<br>(Existence) | 有没有? | `count() > 0` | **Missing** (缺失) |
| **L2** | **完整性**<br>(Completeness) | 全不全? | `active_minutes >= 230`<br>`min_time <= 09:25`<br>`max_time >= 15:00` | **Broken** (断档/早退) |
| **L3** | **准确性**<br>(Accuracy) | 对不对? | `abs(Sum(Vol) - KLine.Vol) < 5%`<br>`Last(Price) == KLine.Close` | **Dirty** (脏数据) |

## 2. 详细校验规则

### 2.1 Level 1: 基础存在性 (Existence Check)
*   **目标**: 快速识别彻底漏采的股票。
*   **输入**: ClickHouse `tick_data_intraday`.
*   **规则**:
    *   如果数据行数为 0，直接判定为 **L1 失败**。
    *   不需要进一步检查 L2/L3。

### 2.2 Level 2: 结构完整性 (Continuity Check)
*   **目标**: 识别网络中断或采集进程崩溃导致的数据残缺。
*   **输入**: Tick 时间戳序列。
*   **规则**:
    1.  **活跃分钟数**: 去重后的分钟数必须 `>= 230` (全天标准交易时长为 240 分钟，容忍 10 分钟停顿)。
    2.  **开盘覆盖**: 首笔时间必须 `<=` 09:25:00 (必须包含集合竞价)。
    3.  **收盘覆盖**: 末笔时间必须 `>= 15:00:00` (必须包含收盘竞价)。
*   **例外**: 若该股票当日发生**全天停牌**或**临时停牌** (由 AI 或公告确认)，则跳过此规则。

### 2.3 Level 3: 数据准确性 (Cross-Validation)
*   **目标**: 识别传输错误、乱序或交易所数据源本身的异常。
*   **输入**: 自采 Tick 数据聚合值 vs 权威日 K 线数据 (Source: TDX/AkShare)。
*   **规则**:
    1.  **成交量对账**: `|Tick总手 - K线总手| / K线总手 <= 0.05` (允许 5% 误差，因统计口径差异)。
    2.  **收盘价对账**: `Tick末笔价格 == K线收盘价` (容忍 0.01 元精度误差)。
*   **判定**: 任一条件不满足，判定为 **L3 失败**。

## 3. 校验输出 (Validation Result)

对于每只股票，校验器将返回如下结构结果：

```json
{
    "code": "600519",
    "status": "FAIL",
    "level": "L3",
    "reasons": ["VolMismatch: Tick=50000 vs KLine=52000 (Diff=3.8%)"],
    "action": "REPAIR"
}
```

## 4. 实现建议
该逻辑应封装在 `TickValidator` 类的 `validate_stock(code, date, kline_ref)` 方法中。
