# 历史数据校验体系落地规划

## 背景与目标

本规划面向个人或小团队的 A 股量化数据系统,假设当前已有日线采集、ODS 入库等基础能力,但**缺少系统性的质量保障机制**。

**目标**:用 4–6 周时间,从"裸奔"状态升级到"具备基本鲁棒性"的数据质量体系;再用 2–3 个月迭代到"可观测、可审计、可自愈"的成熟形态。

**非目标**:不追求一步到位的完美方案;不引入过重的企业级工具(如 Airflow + dbt + Great Expectations 全家桶),保持小团队可维护性。

| **E4** | 跨源比对(L3) | P1 | 1 周 | 双数据源抽样比对 | [E4_跨源比对](./历史数据校验体系/E4_跨源比对.md) |

## 时间线建议

```text
第 1 周:E1 数据模型重构(P0)
第 2 周:E1 收尾 + E2 写入校验
第 3 周:E3 巡检模块
第 4 周:E4 跨源比对
第 5 周:E5 业务规则
第 6 周:E6 补数闭环
第 7 周起:E7 + E8 持续迭代
```

**每周末固定动作**:复盘当周告警、review 误报、调整规则。这个动作至少坚持 3 个月,体系才能从"能用"走到"好用"。

## 风险与避坑提示

| 风险 | 表现 | 应对 |
|---|---|---|
| **过度工程** | 一开始就上 Airflow + dbt + GE | 先用 cron + Python 脚本跑通,跑不动再升级 |
| **告警麻木** | 每天几百条告警没人看 | 严格控制 P0/P1 数量,P2 只入库不推送 |
| **规则爆炸** | 规则越加越多,互相打架 | 定期 review,删除 3 个月没命中过的规则 |
| **修复路径污染** | 自动补数把对的数据改成错的 | 每条规则上线前先在影子模式跑 1 周 |
| **复权事件遗漏** | 漏掉一次分红派息,所有指标错位 | 复权因子对账独立运行,频率高于其他校验 |
| **数据源切换风险** | AkShare 接口变更或限流 | 至少保持双源能力,避免单点依赖 |

---

## 最低可行版(MVP)边界

如果时间紧,**只做 E1 + E2 + E3**(2–3 周)就能解决 70% 的问题:

* E1 让历史数据不再频繁变动
* E2 让脏数据进不来
* E3 让漏采能被发现

这是个人系统的"及格线"。E4–E8 是从 70 分到 95 分的过程,可以根据策略对数据质量的敏感度按需推进。
