# E6: 补数与修复闭环(P1)

> 发现问题必须修复，且修复过程必须可控、可审计、可回滚。

## Story E6-S1: 补数队列设计

* `dq_findings` 表中标记为"需修复"的记录,自动入队 `backfill_queue` (基于 Redis Stream 或数据库任务表实现)
* 队列条目包含:目标表、目标 (ts_code, trade_date)、修复原因、规则版本、优先级、关联的 `request_id`
* 同一 (ts_code, trade_date) 的多次入队自动去重，避免重复处理

## Story E6-S2: 补数执行器

* 异步执行服务 (如依托 `stock-manager-api` 的调度模块)，从队列消费任务
* 限速控制: 每分钟最多处理 N 个任务，结合 `asyncio.Semaphore` 控制并发，避免触发数据源限流 (如 PyWencai 的严格频率限制)
* 优先级调度: 沪深 300 优先于全市场抽样发现的小股票
* **必须幂等**: 同一任务执行 10 次,结果与执行 1 次一致 (使用 `ON CONFLICT DO UPDATE` 机制)
* **资源纪律**: 严格遵守 128MB 内存红线，批量补数采用异步流式处理或分批处理

## Story E6-S3: 审计留痕

* 每次补数操作必须记录到 `backfill_audit`:
* 记录字段：操作时间、目标记录、修复前快照、修复后快照、**追踪的 request_id**
* 触发原因(规则 ID + 规则版本)
* 操作人(自动 / 手动)

## Story E6-S4: 级联失效标记与通知

* 补数完成后，云服务器不负责指标计算，仅根据血缘关系执行失效处理：
* **清理旧数据**: 自动删除或标记失效 (is_valid=0) 受影响的下游派生指标 (如 T 日修复，清理 T 至 T+N 的 MA/MACD)
* **发布信号**: 将需重算的 (ts_code, window_range) 任务发布至 `recalc_signal` 表或消息队列
* **支持外部拉取**: 提供 API 供本地计算节点 (Local Compute Node) 轮询下载待重算任务列表，本地计算完成后再推回云端覆盖存储
* 隔离性: 失效标记操作需极快完成，不阻塞主补数流程

## Story E6-S5: 人工干预通道

* 提供一个简单的命令行 / Web 工具,让人工可以:
* 查看待补数队列及进度
   * 熔断机制：暂停某条规则的自动补数(避免规则 bug 引发大面积非预期更新)
   * 回滚某次错误的修复操作

## Story E6-S6: 异常告警与通知

* 集成系统 `Alerter` 服务，实现关键节点的邮件触达：
* **触发邮件场景**：
   * **自动熔断触发**：当某规则因失败率过高被系统自动熔断时，发送 `CRITICAL` 邮件。
   * **核心失败**：沪深 300 等高优先级标的补数失败超过 3 次。
   * **大范围失效**：单次级联失效影响的记录行数超过阈值（如 > 50,000 行）。
* **通知详情**：邮件必须包含 `request_id`、受影响表名、时间跨度及错误摘要。

## 技术实现细节

### 1. 数据库支撑表 (MySQL)
*   **`backfill_queue`**: 补数任务队列，支持优先级 (Priority 1: HS300) 和幂等去重。
*   **`backfill_audit`**: 审计日志，记录 `before_data` 和 `after_data` (JSON 格式)，支持 `request_id` 全链路追踪。
*   **`recalc_signal`**: 重算信号表，记录需重算的标的及时间范围，供本地计算节点拉取。

### 2. 数据源优先级 (Rule 4)
*   **P0 (Tushare)**: 主力补数源，负责 K 线及复权因子修复。
*   **P1 (BaoStock)**: 辅助补数源，用于 Tushare 积分受限时的降级。
*   **P2 (AkShare)**: 财务数据及事件类数据补数。

### 3. 级联失效机制
*   基于 `lineage.yaml` 定义数据血缘。
*   当 ODS 层数据修复后，自动清理 ADS 层受影响的区间数据（如 MA/MACD 等滑动窗口指标）。
*   同步发射重算信号，确保数据最终一致性。

## 验收标准

* [x] 制造一个数据缺失,从检测发现到补数修复的全链路自动跑通
* [x] 修复后下游 MA60、MACD 等指标在云端被正确清理并标记失效
* [x] 验证 `recalc_signal` 能够正确生成供外部拉取的重算指令数据
* [x] 每次修复均带有 `request_id` 并在 audit 表中可追溯
* [x] 验证触发熔断时，`Alerter` 能够正确发送告警邮件至预设邮箱
