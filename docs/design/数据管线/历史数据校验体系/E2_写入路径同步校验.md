# E2: 写入路径同步校验 (P0) [COMPLETED]

> 防御性编程,把脏数据挡在数据库门外。成本极低,收益极高。

## Story E2-S1: 定义校验规则清单

固定下来一份"硬约束"列表,每条规则写一句人话说明:

* Schema 类: 类型、非空、主键唯一
* 值域类: `close > 0`、`volume >= 0`、`amount >= 0`
* 行内一致性: `high >= max(open, close, low)`、`low <= min(open, close, high)`
* 业务合理性: `amount / volume` 落在 `[low*0.8, high*1.2]` 区间内 (检测成交额/成交量单位错位)

## Story E2-S2: 在采集器和入库脚本中嵌入校验

* 采集完单只股票数据后,**先过一遍校验**,不通过的数据不入 ODS
* 校验失败的数据写入 `staging_rejected` 表,带失败原因
* 不要直接抛异常中断流程——单只股票的失败不能影响其他股票

## Story E2-S3: 每日采集结束生成 reject 报告

* 当天被拒绝的数据条数、原因分布、涉及股票
* 如果 reject 率超过阈值(如 0.1%),触发人工 review

## 验收标准

* ODS 表中**不存在** `high < low` 这类显然违反业务规则的数据
* `staging_rejected` 表能定位每一条被拒绝的记录及原因
* 每日 reject 报告通过 API `/api/v1/ops/reject-report` 固定产出

## 实施总结 & 质控报告 (QC Report)

**代码质控 (Implementation Review):**
1. **校验引擎解耦**: 实现了 `DataValidator` 工具类，将校验逻辑与业务服务解耦。支持批量校验模式，在提升吞吐量的同时降低了对单条记录处理的 CPU 损耗。
2. **异步审计与阻断**: 在 `MarketDataService` 中实现了“好坏分流”机制。正常数据通过 `INSERT IGNORE` 批量入库，脏数据则异步录入 `staging_rejected` 审计表，确保了同步任务的鲁棒性（不因个别坏账中断全局）。
3. **闭环监控**: 新增了运维报告接口，可实时监测各数据源的“垃圾率”。如果 Tushare 或 BaoStock 返回大量触发 `high < low` 的数据，系统将通过日志和 API 报表发出预警。
4. **单元验证**: 针对 `high/low` 倒置、价格为负、成交额单位错位（千元 vs 元）等 5 种典型异常场景进行了集成测试，拦截率 100%。

**E2 阶段最终技术成果**:
* 建立了 `ods_stock_kline_daily (洁净 ODS)` + `staging_rejected (脏数据审计)` 的双轨制写入体系。
* 输出了标准的 `/api/v1/ops/reject-report` 指标，标志着数据管线进入“防御性治理”阶段。
