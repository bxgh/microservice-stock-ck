# E1: 数据模型重构 (P0) [COMPLETED]

> 不重构数据模型,后面所有校验都是治标不治本。这一步省不掉。

## Story E1-S1: 盘点现有表结构

* 列出当前所有日线 / 分钟线 / 复权相关表
* 标记每张表的"可变性":是 append-only 还是会被更新
* 识别"被反复刷写"的表(通常就是前复权 K 线)

## Story E1-S2: 设计三层数据模型

* **ODS 层**:只存原始未复权数据,主键 `(code, date)`,**写入后永不修改**
* **维度层**:独立的 `stock_adjust_factor` 表,存复权因子时间序列,append-only
* **DWD/视图层**:前复权、后复权按需通过 SQL 视图或物化视图计算

## Story E1-S3: 迁移历史数据

* 从现有表反算出原始 OHLC,写入新 ODS 表
* 从权威源(如交易所公告 / Tushare 的 adj_factor 接口)拉一份完整复权因子序列
* **双写阶段**:新旧表并存 1–2 周,跑差异比对,确认无偏差后切换

## 验收标准

* 任意一只股票的任意历史日期,在 ODS 表中的 OHLC **永远等于交易所原始数据**
* 前复权 / 后复权结果通过视图查询,响应时间在可接受范围内(可加物化视图优化)
* 一次分红派息事件,**只新增 1 行复权因子记录**,不触发任何历史数据写入

## 实施总结 & 质控报告 (QC Report)

**代码质控 (Git Diff Review):**
1. **数据写入幂等性**: `ods_stock_factor_daily` (连续因子缓存表) 的写入逻辑已由 `INSERT IGNORE` 变更为 `INSERT INTO ... ON DUPLICATE KEY UPDATE`。作为衍生层数据，这赋予了其“自我修复”的能力。当历史分红数据发生回溯修正时，系统能在下一次同步中自动覆盖旧错误因子，同时严格保持底层 ODS 原始数据的不可篡改性。
2. **触发机制前置**: 缓存表的刷新触发点 (`_refresh_factor_cache`) 已从“因子同步后”重构为“K 线同步后”。这精准解决了在无分红的常规交易日中，衍生表不生成记录从而导致视图 JOIN 缺失（行情断档）的系统性 bug。
3. **计算逻辑下推与隔离**: `IndicatorService` 全面切换至 `v_stock_kline_forward_adj` 视图进行读取。L2 级别的二次计算（如行业广度、热点领涨股统计）彻底隔离了由除权除息带来的价格跳空污染。
4. **数据库环境适配**: 考虑到生产环境采用的是 MySQL 5.7，已将高性能的 `ROW_NUMBER() OVER()` 窗口函数批量查询逻辑，降级为高度兼容的**关联子查询 (Correlated Subquery)** 模式，确保系统稳定无报错。

**E1 阶段最终技术成果**:
* 构建了 `原始 K 线 (不可变)` + `分红事件 (不可变)` + `连续因子缓存 (自我修正)` + `复权视图 (实时计算)` 的闭环防御体系。
* 面向前端输出了标准化的 `/api/v1/market/kline` 高级行情接口，使应用层代码无需关注复杂的复权算法。
