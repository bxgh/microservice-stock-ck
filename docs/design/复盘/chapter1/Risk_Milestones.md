# 第 1 章 · 风险、规避与里程碑

## 风险与避坑

### 数据采集类

1. **Tushare `pct_chg` 单位陷阱**:Tushare 返回的是百分比(1.23 = 1.23%),入库前必须 `/ 100` 转为小数,否则后续所有相对比较会差 100 倍
2. **akshare 涨跌停池接口字段不稳定**:东财公开接口字段名偶有变化,采集层必须做字段映射 + 单元测试,不要硬编码字段名
3. **代码后缀补齐**:akshare 返回的代码可能不带 `.SH/.SZ`,入库前必须根据代码前缀补齐(`6/9/688/5xx → .SH`,`0/3/1xx/159 → .SZ`,`8/4 → .BJ`)
4. **`899050.BJ` 北证 50 数据源**:Tushare 是否完整覆盖北证 50 历史需验证,缺失则用 akshare 兜底
5. **`932000.CSI` 中证 2000 代码**:中证系列指数代码格式 Tushare 用 `.CSI` 后缀,验证实际可用代码
6. **万得全 A 代码**:Tushare 中的可用代码需要验证,标记为 TBD

### 数据质量类

7. **样本范围一致性**:`up_count + down_count + flat_count` 必须基于同一过滤口径(剔除 B 股 / 停牌 / 新股),不一致会导致涨跌比失真
8. **新股次新股污染连板**:上市 < 60 日的新股不应纳入连板池,否则会出现"假高板",计算 `board_height` 时必须剔除
9. **ST 股涨跌停**:ST ±5%、非 ST ±10/20/30%,`limit_up_count` 是否包含 ST 需口径明确(建议**包含**,但单独有 `st_limit_up_count` 字段在 L4 完整版中区分)
10. **节假日处理**:务必使用 `trade_cal` 而非自然日历,否则月末 / 季末跨节日会出现空数据

### 指标计算类

11. **`turnover_pctile_1y` 至少需要 250 个交易日历史**:历史回补不足时,该字段返回 NULL 而非 0,前端需做空值处理
12. **`market_regime` 阈值需校准**:首版用经验值(`turnover_pctile < 0.20` / `up_down_ratio > 3 且 limit_up > 60`),建议运行 3 个月后基于历史分布重新校准
13. **`max_board_height` 异常值**:个别股票"伪连板"(如重组复牌后连续涨停),需要剔除停牌时长 > 30 日的股票
14. **`turnover_total` 跨 K 线源一致性**:本章用 `stock_kline_daily.amount` 求和,若与 `daily_info`(交易所披露口径)对比有差异,以 `daily_info` 为准做校验,差异 > 1% 告警

### 工程类

15. **指数日线 8000 行限制**:Tushare `index_daily` 单次最多 8000 行,**全市场指数 + 5 年历史会超**,采集脚本需按代码循环或按日期分段
16. **L1 计算 SQL 性能**:涨跌家数 / 成交额聚合在 1800 万行的 `stock_kline_daily` 上跑,必须用 `(trade_date, code)` 索引,且按日切片计算,避免全表扫描
17. **首次回补 SQL 限制**:`ads_l1_market_overview` 历史回补不能用上述单日 SQL 跑 1200 次,Antigravity 需要写循环或基于 `trade_cal` 批量执行,建议每次回补 1 个月
