# MySQL 5.7 完整建表 DDL

按你的决策落地:

1. ✅ V1 仅支持 Markdown 导出
2. ✅ 公众号发布支持二次编辑(`mp_publish_record` 存独立快照)
3. ✅ `wx_user.status` 预留封禁/注销
4. ✅ **日记相关表统一前缀 `diary_`**

## 表名前缀规范

| 域 | 前缀 | 表 |
|---|---|---|
| 用户系统 | `sys_` | `sys_user` |
| 日记主域 | `diary_` | `diary_entry` / `diary_stock` / `diary_tag` / `diary_tag_dict` / `diary_attachment` / `diary_export_task` |
| 公众号域 | `mp_` | `mp_account` / `mp_publish_record` / `mp_media_cache` |
| 股票主数据 | `stock_` | `stock_info` |

**说明**:
- 原 `wx_user` → `sys_user`,因为它是全系统根表,不应带 `wx_` 这种渠道前缀(未来可能支持 App Web 登录)
- 原 `tag` → `diary_tag_dict`,标签字典明确归属日记域;关联表 `diary_tag`(diary 与 tag 的关联)
- `stock_info` 暂不动,等股票主数据相关表多了再考虑加 `stock_` 前缀域

## 几个关键设计点说明

### 1. 没有外键约束

DDL 里所有 `xxx_id` 都没声明 `FOREIGN KEY`,这是有意的:

| 不加外键的理由 | 说明 |
|---|---|
| 性能 | 外键约束在高并发下有锁开销 |
| 灵活性 | 软删除场景下,被引用记录"逻辑不存在物理仍在",外键反而碍事 |
| 微服务演进 | 未来拆库后,跨库外键无法维护 |
| 行业惯例 | 大型互联网项目通常**应用层保证完整性**,而非数据库 |

**应用层保证**:Service 层在创建/删除关联时,显式校验目标记录存在。

### 2. 软删除字段名 `deleted_at`,不是 `is_deleted`

之前讨论时用 `is_deleted TINYINT`,这版改为 `deleted_at DATETIME`:

| 优势 | 说明 |
|---|---|
| 信息更丰富 | 同时记录"是否删除"和"何时删除" |
| 90 天自动清理 | `WHERE deleted_at < NOW() - INTERVAL 90 DAY` 直接查 |
| 行业主流 | Laravel、Django 等 ORM 默认用此约定 |
| 查询稍冗长 | 改用 `WHERE deleted_at IS NULL`,可接受 |

### 3. 全文索引 `ngram` 解析器

`diary_entry` 的 `ft_content`:
```sql
FULLTEXT KEY `ft_content` (`title`, `content`) WITH PARSER ngram
```

MySQL 5.7 内置 `ngram` 解析器,默认 2-gram(`ngram_token_size=2`),对中文搜索够用。

**注意**:用前需在 `my.cnf` 确认 `innodb_ft_min_token_size=1` 或保持默认 3,影响搜索结果。

查询示例:
```sql
SELECT id, title, excerpt 
FROM diary_entry 
WHERE user_id = 123 
  AND deleted_at IS NULL
  AND MATCH(title, content) AGAINST('茅台' IN NATURAL LANGUAGE MODE)
ORDER BY entry_date DESC LIMIT 20;
```

### 4. 索引设计原则落地

最常见查询场景对应的索引:

| 场景 | 索引 |
|---|---|
| 用户日记列表(默认按日期倒序) | `idx_user_date (user_id, entry_date DESC)` |
| 按类型筛选 | `idx_user_type (user_id, entry_type)` |
| 置顶日记优先 | `idx_user_pinned (user_id, is_pinned, entry_date DESC)` |
| 某股票的所有日记 | `diary_stock.idx_stock_diary` |
| 全文搜索 | `ft_content` |
| 标签下的日记 | `diary_tag.idx_tag_diary` |

**没建索引的字段**:
- `mood`、`visibility`:选择性低(每个值都覆盖大量数据),建索引意义不大
- `is_deleted` 同理(99% 都是未删除),不单独建,而是依赖应用层 `WHERE deleted_at IS NULL` + 其他条件的联合索引覆盖

### 5. JSON 字段使用约定

每个主表都有 `meta JSON DEFAULT NULL`,作为**临时扩展位**。规则:

- 不在 JSON 里存需要查询/过滤/排序的字段
- 新业务字段先放 JSON 试用,稳定后再 alter table 提为正式字段
- 应用层做版本兼容(读 JSON 时容忍字段缺失)

**`prefs JSON` 是例外**:它就是设计用来存配置的,长期就放 JSON 里。

### 6. 字符长度选择

| 字段 | 长度 | 理由 |
|---|---|---|
| `openid` / `unionid` | 64 | 微信文档说 28-32 字符,留余量 |
| `cos_key` | 255 | 路径含日期、用户 ID、UUID,够用 |
| `excerpt` | 255 | 60 个汉字 = 180 字节,留余量 |
| `error_message` | 512 | 截断长错误信息 |
| `content` | MEDIUMTEXT(16MB) | 不可能写满,但避免 TEXT(64KB) 限制 |
| `content_html`(发布) | MEDIUMTEXT | 微信图文上限 20000 字,HTML 可能膨胀 |
